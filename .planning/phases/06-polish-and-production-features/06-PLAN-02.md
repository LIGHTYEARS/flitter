# Plan 02: Mouse Support and Error Boundaries

<objective>
Enable mouse wheel scrolling in chat view and add error boundary rendering to display errors gracefully in the TUI without crashing.
</objective>

<must_haves>
- Mouse wheel scrolls chat view
- Errors display in TUI as red text, not crashes
- App remains functional after errors
</must_haves>

<tasks>

<task id="02.1">
<title>Add error display to ChatView</title>

<read_first>
- packages/flitter-amp/src/widgets/chat-view.ts (current build method, item rendering)
- packages/flitter-amp/src/state/app-state.ts (error field, setError, clearError)
</read_first>

<action>
Modify `packages/flitter-amp/src/widgets/chat-view.ts`:

The ChatView currently renders ConversationItem[]. Add support for an optional `error` prop that renders an error banner at the top of the chat when present.

1. Add `error?: string | null` to ChatViewProps
2. In the build method, if `error` is truthy, prepend an error widget to the children:
```typescript
if (this.error) {
  children.unshift(
    new Container({
      padding: EdgeInsets.symmetric({ horizontal: 2, vertical: 1 }),
      child: new Text({
        text: new TextSpan({
          text: `Error: ${this.error}`,
          style: new TextStyle({ foreground: Color.red, bold: true }),
        }),
      }),
    })
  );
}
```

3. In `app.ts`, pass `appState.error` to ChatView:
```typescript
new ChatView({ items, error: appState.error }),
```
</action>

<acceptance_criteria>
- packages/flitter-amp/src/widgets/chat-view.ts contains `error` in props
- packages/flitter-amp/src/widgets/chat-view.ts contains `Color.red` for error display
- packages/flitter-amp/src/app.ts contains `error: appState.error`
- `npx tsc --noEmit --skipLibCheck` passes
</acceptance_criteria>
</task>

<action>
Mouse support in flitter-core is typically enabled at the WidgetsBinding level. Check if `runApp` options support mouse enabling. If the framework handles mouse events automatically when enabled, we just need to ensure:

1. The `runApp` call in `startTUI` passes mouse: true (if such option exists)
2. ScrollController already supports programmatic scrolling — mouse wheel events would be handled by SingleChildScrollView if the framework dispatches them

If flitter-core does NOT have built-in mouse support, create a placeholder comment in app.ts:
```typescript
// TODO: Mouse support requires flitter-core mouse event dispatch
// ScrollController.scrollBy() is available for programmatic scroll
// Mouse click positioning requires TextField cursor API
```

The key insight: mouse support depends on what flitter-core provides. If the framework dispatches scroll events to SingleChildScrollView, it works automatically. If not, this is deferred.
</action>
