### 2. connection.ts — MCPConnection 连接生命周期

```ts
// ─── Client 类 (transport wrapper) ────────────────────
// 包装 transport + RequestManager，提供高层 API
class MCPClient {
  constructor(clientInfo: ImplementationInfo, capabilities: ClientCapabilities);

  // 连接: transport.start() → send initialize → receive result → send notifications/initialized
  async connect(transport: MCPTransport, opts?: { timeout?: number }): Promise<void>;

  // 高层 API (通过 RequestManager 发送 JSON-RPC)
  async listTools(): Promise<{ tools: MCPTool[] }>;
  async callTool(params: { name: string; arguments?: Record<string, unknown> }, signal?: AbortSignal): Promise<MCPToolResult>;
  async listResources(): Promise<{ resources: MCPResource[] }>;
  async readResource(uri: string, signal?: AbortSignal): Promise<{ contents: unknown[] }>;
  async listPrompts(): Promise<{ prompts: MCPPrompt[] }>;
  async getPrompt(params: { name: string; arguments?: Record<string, unknown> }, signal?: AbortSignal): Promise<unknown>;

  // 通知处理
  setNotificationHandler(method: string, handler: () => void): void;
  removeNotificationHandler(method: string): void;

  getServerCapabilities(): ServerCapabilities | undefined;
  getServerVersion(): ImplementationInfo | undefined;

  async close(): Promise<void>;
}

// ─── 重连参数 ─────────────────────────────────────────
const RECONNECT_CONFIG = {
  initialDelayMs: 1000,
  backoffFactor: 2,
  maxDelayMs: 30000,
  maxRetries: 5,
};

// 永久错误码 (不重连)
const PERMANENT_ERROR_CODES: number[];

// ─── MCPConnection ──────────────────────────────────
export interface MCPConnectionStatus {
  type: "connecting" | "authenticating" | "connected" | "reconnecting"
       | "denied" | "awaiting-approval" | "failed";
  capabilities?: ServerCapabilities;
  serverInfo?: ImplementationInfo;
  error?: { code: number; message: string };
  attempt?: number;
  nextRetryMs?: number;
}

export class MCPConnection {
  readonly status$: BehaviorSubject<MCPConnectionStatus>;
  readonly tools$: BehaviorSubject<MCPTool[]>;
  readonly resources$: BehaviorSubject<MCPResource[]>;
  readonly prompts$: BehaviorSubject<MCPPrompt[]>;
  readonly toolsLoaded$: BehaviorSubject<boolean>;

  constructor(
    spec: MCPServerSpec,
    opts: {
      workingDirectory?: string;
      loadProfile?: string;
      authProvider?: MCPAuthProvider;
      serverName?: string;
    }
  );

  // 高层 API (delegate to active client)
  async callTool(params: { name: string; arguments?: Record<string, unknown> }, signal?: AbortSignal): Promise<MCPToolResult>;
  async listResources(signal?: AbortSignal): Promise<MCPResource[]>;
  async readResource(uri: string, signal?: AbortSignal): Promise<unknown[]>;
  async getPrompt(name: string, args?: Record<string, unknown>, signal?: AbortSignal): Promise<unknown>;

  async dispose(): Promise<void>;
}
```

**MCPConnection 内部流程:**

1. **创建 transport**: 根据 spec 类型选择
   - `"command" in spec` → `new StdioTransport({ command, args, env, cwd })`
   - `"url" in spec` → 先尝试 `StreamableHTTPTransport`，失败回退到 `SSETransport`

2. **连接**: `client.connect(transport)` → initialize handshake

3. **工具/资源/提示发现**: 连接成功后 `listTools()` / `listResources()` / `listPrompts()`

4. **通知监听**: 注册 `notifications/tools/list_changed` 等 handler → 重新 list → 更新 Observable

5. **重连**: transport close → 调度重连 (指数退避) → 重新创建 transport + connect → 重新发现工具

6. **销毁**: 取消重连定时器 + close client

### 3. server-manager.ts — MCPServerManager 服务器管理器

```ts
export interface MCPServerInfo {
  name: string;
  spec: MCPServerSpec;
  status: MCPConnectionStatus;
  tools: MCPTool[];
  resources: MCPResource[];
  prompts: MCPPrompt[];
}

export interface NamespacedMCPTool {
  name: string;            // mcp__server__tool 格式
  originalName: string;    // 原始工具名
  serverName: string;      // 所属 server
  description?: string;
  inputSchema: Record<string, unknown>;
}

export class MCPServerManager {
  readonly servers$: BehaviorSubject<Map<string, MCPServerInfo>>;
  readonly allTools$: BehaviorSubject<NamespacedMCPTool[]>;

  constructor(opts: {
    getConfig: () => { mcpServers?: Record<string, MCPServerSpec> };
    workingDirectory?: string;
    createOAuthProvider?: (serverName: string, url: string) => MCPAuthProvider | undefined;
  });

  // 手动刷新 (重新读取配置)
  refresh(): void;

  // 按命名空间工具名调用
  async callTool(
    namespacedName: string,
    args: Record<string, unknown>,
    signal?: AbortSignal
  ): Promise<MCPToolResult>;

  async dispose(): Promise<void>;
}
```

**MCPServerManager 内部流程:**

1. **初始化**: 读取 `mcpServers` 配置 → 为每个 server 创建 `MCPConnection`
2. **配置变更**: 对比新旧 spec (JSON.stringify 比较)，只有变更的 server 才重建
3. **工具聚合**: 合并所有 server 的 `tools$` → 加上命名空间前缀 → 更新 `allTools$`
4. **工具调用路由**: 解析 `mcp__server__tool` → 找到对应 MCPConnection → `callTool(originalName, args)`
5. **销毁**: dispose 所有 MCPConnection
