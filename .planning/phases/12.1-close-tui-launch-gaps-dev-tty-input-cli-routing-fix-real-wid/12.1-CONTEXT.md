# Phase 12.1: Close TUI Launch Gaps - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the three-layer gap preventing `flitter` CLI from launching a working interactive TUI session. Layer 1: TuiController uses process.stdin instead of /dev/tty. Layer 2: resolveCliContext TTY logic diverges from amp's S8(). Layer 3: Application-layer widgets return placeholders and ConversationView isn't wired into the widget tree. This phase closes ALL gaps — no deferral to future phases. End state: `flitter` launches a real TUI with visible conversation UI, verified via tmux-capture.

</domain>

<decisions>
## Implementation Decisions

### Terminal Input Strategy
- **D-01:** Port amp's /dev/tty strategy exactly — open `/dev/tty` via `fs.openSync` for input, wrap in `tty.ReadStream`, call `setRawMode(true)`. Output uses `process.stdout` with fallback to `/dev/tty` write. Matches amp's `eXT()`/`Zu0()`/`hy0()` functions.
- **D-02:** Port amp's Bun version check from `Qu0()` — Bun < 1.2.22 falls back to `process.stdin`. Exact version boundary preserved.
- **D-03:** Port amp's signal handler chain for /dev/tty cleanup — graceful SIGINT/SIGTERM handling with alt screen exit, raw mode restore, and ReadStream close.

### CLI Mode Routing
- **D-04:** Match amp's S8() logic exactly for `resolveCliContext`: `executeMode = !!opts.execute || (!process.stdout.isTTY && !opts.streamJson)`. isTTY checks stdout only, not stdin. No `--interactive` override flag.

### Widget Tree Composition
- **D-05:** Port amp's `_70()` widget tree exactly for `interactive.ts`: `AppWidget → ThemeController → ConfigProvider → ConversationLayout` with `ConversationView` (scrollable) + `InputField` (pinned bottom) + status bar.
- **D-06:** Port amp's full layout from `_70()` — includes separator lines, status bar, proper vertical space distribution between conversation and input areas.
- **D-07:** Port amp's status bar showing model name, token count, thread info.

### Scrolling
- **D-08:** Port amp's scroll strategy directly from `tui-widget-framework.js` for ConversationView scrolling.
- **D-09:** Port amp's scroll RenderObject — rewrite Scrollable as a proper RenderBox subclass with standard `performLayout(BoxConstraints)` connected to PipelineOwner layout pipeline.

### Reactive Data Flow
- **D-10:** Port amp's reactive pattern from `html-sanitizer-repl.js` / `k40()` — how ThreadWorker events connect to widget state and trigger rebuilds.
- **D-11:** Port amp's streaming display approach — how SSE deltas propagate through the widget tree and get rendered frame-by-frame.

### Error Handling
- **D-12:** Port amp's error display — inline error message in conversation view with retry option, red-highlighted text block.

### Thread Initialization
- **D-13:** Port amp's thread init flow — check for `--continue` flag, show thread picker if multiple exist, create new thread with welcome message otherwise.

### Visual Fidelity
- **D-14:** Full visual fidelity for all widget `build()` methods: ConversationView renders markdown via micromark (code blocks, inline formatting, role indicators), InputField shows cursor/border/placeholder, status bar with model + token count, proper padding/spacing.

### Bun Compatibility
- **D-15:** Fix the 9 failing TuiController tests due to Bun's `process.stdin.unref()` incompatibility. Port amp's Bun compatibility handling.

### Verification
- **D-16:** Both automated tmux-capture visual regression tests AND human UAT checklist. tmux-capture tooling already exists in the project.

### Claude's Discretion
- Exact ANSI color codes for error highlighting
- Welcome message text content
- Thread picker UI layout details (if interactive picker is complex, simplify to flag-based resume)

</decisions>
