# Phase 12.1: Close TUI Launch Gaps - Research

**Researched:** 2026-04-15
**Domain:** Terminal I/O, CLI routing, Widget tree composition, reactive state binding
**Confidence:** HIGH

## Summary

Phase 12.1 closes three interconnected gaps preventing `flitter` CLI from launching a working interactive TUI session. The research covers: (1) /dev/tty-based terminal input to replace the current process.stdin approach, (2) CLI mode routing alignment with amp's S8() function, (3) full widget tree composition with real rendering, and (4) reactive data flow from ThreadWorker events to widget rebuilds.

The implementation is a faithful port from amp's reverse-engineered source. All canonical references have been read and the patterns are well-understood. The existing flitter infrastructure (three-tree engine, layout widgets, markdown renderer, scroll controller, text editing, focus management) provides a solid foundation -- the main work is wiring these pieces together correctly and replacing placeholder `build()` methods with real widget trees.

**Primary recommendation:** Execute as a layered port -- fix /dev/tty input first (enables all other testing), then CLI routing, then widget tree composition, then reactive data flow. Each layer builds on the previous and can be independently verified.

## Architecture Patterns

### Layer 1: /dev/tty Input Strategy

The current `TuiController.init()` uses `process.stdin` directly (line 181-193 of tui-controller.ts). This is wrong because when the CLI is invoked via pipe or subprocess, `process.stdin` may be redirected. Amp solves this with a two-tier strategy: [VERIFIED: amp-cli-reversed/framework/tui-widget-framework.js lines 768-883]

**Zu0() -- /dev/tty strategy (primary):**
```typescript
// Source: amp's Zu0() in tui-widget-framework.js:768-813
// 1. Open /dev/tty directly for reading
const fd = fs.openSync("/dev/tty", "r");
// 2. Verify it's actually a TTY
if (!tty.isatty(fd)) throw new Error("/dev/tty is not a TTY device");
// 3. Create ReadStream wrapper
const stream = new tty.ReadStream(fd);
// 4. Enable raw mode on the tty stream (not process.stdin)
stream.setRawMode(true);
// 5. Attach data handler
stream.on("data", (data) => { /* feed to InputParser */ });
```

**Ju0() -- process.stdin fallback (Bun < 1.2.22 or no /dev/tty):**
```typescript
// Source: amp's Ju0() in tui-widget-framework.js:815-879
// Falls back to process.stdin when /dev/tty is unavailable
const stdin = process.stdin;
if (stdin.isTTY) stdin.setRawMode(true);
stdin.on("data", (data) => { /* feed to InputParser */ });
```

**eXT() -- router:**
```typescript
// Source: amp's eXT() in tui-widget-framework.js:881-883
function createTtyInput(): TtyInputSource {
  if (isBunWithTtyBug()) return createStdinFallback(); // Ju0
  return createDevTtyInput(); // Zu0
}
```

**Qu0() -- Bun version check:**
```typescript
// Source: amp's Qu0() in tui-widget-framework.js:679-689
// Returns true if running Bun < 1.2.22 (has /dev/tty bugs)
function isBunWithTtyBug(): boolean {
  const version = "1.3.10"; // Bun.version - hardcoded in reversed code
  if (!version) return false;
  const [major, minor, patch] = version.split(".").map(Number);
  if (major !== 1 || minor !== 2) return false;
  return patch < 22;
}
```

**hy0() -- output strategy:**
```typescript
// Source: amp's hy0() in tui-widget-framework.js:903-934
function createTtyOutput(): TtyOutputTarget {
  if (process.stdout.isTTY) return { stream: process.stdout, target: "stdout", dispose: () => {} };
  if (process.stderr.isTTY) return { stream: process.stderr, target: "stderr", dispose: () => {} };
  // Last resort: open /dev/tty for writing
  try {
    const fd = fs.openSync("/dev/tty", "w");
    if (tty.isatty(fd)) return {
      stream: { write(data) { fs.writeSync(fd, data); return true; } },
      target: "dev-tty",
      dispose: () => { fs.closeSync(fd); }
    };
    fs.closeSync(fd);
  } catch {}
  return { stream: process.stdout, target: "stdout", dispose: () => {} };
}
```

**Key architectural pattern -- TtyInputSource interface:**
```typescript
// Interface matching amp's Zu0/Ju0 return type
interface TtyInputSource {
  stdin: tty.ReadStream | NodeJS.ReadStream | null;
  dataCallback: ((data: Buffer) => void) | null;
  earlyInputBuffer: Buffer[];
  init(): void;
  on(event: "data", handler: (data: Buffer) => void): void;
  pause(): void;
  resume(): void;
  dispose(): void;
  getEarlyInputText(): string;
}
```
[VERIFIED: amp-cli-reversed/framework/tui-widget-framework.js]

**Early input buffering (P8 pattern):** Amp buffers any stdin data that arrives before the TUI is ready, via a module-level `P8` object. The `getEarlyInputText()` method drains the buffer and strips ANSI escape sequences (via `aXT()`). This handles the case where the user types before the TUI finishes initialization. ### Layer 2: CLI Mode Routing Fix

**Current bug in resolveCliContext:**
```typescript
// Current (WRONG): packages/cli/src/context.ts line 62
const executeMode = Boolean(opts.execute) || !isTTY || Boolean(opts.headless);
// Where isTTY = Boolean(process.stdout.isTTY && process.stderr.isTTY)
```

This is wrong in two ways:
1. It checks `!isTTY` which is `!(stdout.isTTY && stderr.isTTY)` -- if stderr is not TTY, it wrongly enters execute mode
2. It doesn't account for `--stream-json` flag suppressing execute mode

**Correct logic from amp's S8():**
```typescript
// Source: amp's S8() in cli-entrypoint.js:922-923
const executeMode = !!opts.execute || (!process.stdout.isTTY && !opts.streamJson);
const isTTY = process.stdout.isTTY && process.stderr.isTTY;
```
[VERIFIED: amp-cli-reversed/app/cli-entrypoint.js lines 922-923]

Key differences:
- Execute mode checks **only stdout** TTY status, not stderr
- `--stream-json` flag **prevents** execute mode (it's a streaming output mode, not batch)
- `isTTY` is a separate flag for "both stdout and stderr are TTY" (used for output formatting)

## Common Pitfalls

### Pitfall 1: process.stdin vs /dev/tty confusion
**What goes wrong:** When `process.stdin` is piped (e.g., `echo "hello" | flitter`), it's not a TTY and `setRawMode()` throws. Even when stdin is a TTY, it may be shared with other processes.
**Why it happens:** Node.js process.stdin is the standard input stream which can be redirected. `/dev/tty` is the actual controlling terminal.
**How to avoid:** Always open `/dev/tty` directly via `fs.openSync("/dev/tty", "r")` and create a dedicated `tty.ReadStream`. Only fall back to process.stdin for Bun < 1.2.22 compatibility.
**Warning signs:** `ENOTTY` errors, `setRawMode is not a function`, stdin not receiving keystrokes.

### Pitfall 2: Forgetting to dispose /dev/tty file descriptor
**What goes wrong:** Leaked file descriptors prevent clean process exit, or subsequent runs fail with "too many open files."
**Why it happens:** `fs.openSync()` returns a raw fd that must be explicitly closed. If deinit() doesn't call `stream.destroy()` or the process crashes, the fd leaks.
**How to avoid:** Always close in `dispose()`: `stream.setRawMode(false); stream.removeAllListeners("data"); stream.destroy();`. Register cleanup in signal handlers.
**Warning signs:** Process hangs on exit, file descriptor count grows over time.

### Pitfall 3: executeMode routing check on wrong stream
**What goes wrong:** CLI enters execute mode when it should be interactive (or vice versa), because the TTY check looks at the wrong stream.
**Why it happens:** Current code checks `!isTTY` where `isTTY = stdout && stderr`. Amp only checks `!process.stdout.isTTY` for executeMode.
**How to avoid:** Follow amp's S8() exactly: `executeMode = !!opts.execute || (!process.stdout.isTTY && !opts.streamJson)`.
**Warning signs:** `flitter` launches in execute mode in normal terminal, or stays interactive when piped.

### Pitfall 4: Widget build() returning wrong tree shape after setState
**What goes wrong:** ConversationView or ThreadStateWidget returns a different Widget type on rebuild, causing the Element tree to tear down and recreate instead of update in place.
**Why it happens:** Flutter's three-tree reconciliation matches widgets by `runtimeType` and `key`. If build() returns `Column` on first call but `Text` on error state, the entire subtree is unmounted and remounted.
**How to avoid:** Always return the same top-level Widget type from build(). Use conditional children within the same parent type.
**Warning signs:** Flicker on state changes, scroll position resets, focus loss.

### Pitfall 5: Scrollable RenderBox not requesting layout on scroll
**What goes wrong:** Scrolling changes the scroll offset but nothing repaints because the render pipeline doesn't know the offset changed.
**Why it happens:** ScrollController.jumpTo() updates the offset but the RenderBox's `markNeedsPaint()` isn't called.
**How to avoid:** RenderScrollable must listen to ScrollController and call `markNeedsPaint()` (not layout -- scroll offset changes only affect paint, not layout).
**Warning signs:** Scroll keys work (controller.offset changes) but screen doesn't update.

### Pitfall 6: Bun process.stdin.unref() incompatibility
**What goes wrong:** Tests hang because Bun's process.stdin doesn't support `.unref()`.
**Why it happens:** Current TuiController calls `process.stdin.unref()` on non-TTY environments (line 196). Bun's stdin implementation differs from Node.js.
**How to avoid:** With /dev/tty strategy, the non-TTY stdin fallback path is only used for Bun < 1.2.22. For the primary /dev/tty path, there's no need to unref -- the stream is owned and can be destroyed directly. For test compatibility, guard `.unref()` with a try/catch.
**Warning signs:** Test process never exits, "unref is not a function" error.

### Pitfall 7: Circular subscription in reactive flow
**What goes wrong:** ThreadStateWidget subscribes to ThreadStore in initState(), but setState() during subscription callback causes immediate rebuild, which calls initState() again if the widget is recreated.
**Why it happens:** If the widget tree structure changes during rebuild (see Pitfall 4), a new State is created and subscriptions double up.
**How to avoid:** Use `didUpdateWidget()` to transfer subscriptions when the widget updates rather than recreating. Store subscription reference and always unsubscribe in `dispose()`.
**Warning signs:** Memory leak, exponential rebuild cascades, duplicate messages in view.

### /dev/tty Input Source Implementation

```typescript
// Source: amp's Zu0() in tui-widget-framework.js:768-813
import * as fs from "node:fs";
import * as tty from "node:tty";

interface TtyInputSource {
  stdin: tty.ReadStream | NodeJS.ReadStream | null;
  dataCallback: ((data: Buffer) => void) | null;
  earlyInputBuffer: Buffer[];
  init(): void;
  on(event: "data", handler: (data: Buffer) => void): void;
  pause(): void;
  resume(): void;
  dispose(): void;
  getEarlyInputText(): string;
}

function createDevTtyInput(): TtyInputSource {
  let drainedEarly = false;
  const source: TtyInputSource = {
    stdin: null,
    dataCallback: null,
    earlyInputBuffer: [],
    init() {
      if (this.stdin !== null) return;
      const fd = fs.openSync("/dev/tty", "r");
      if (!tty.isatty(fd)) throw new Error("/dev/tty is not a TTY device");
      const stream = new tty.ReadStream(fd);
      this.stdin = stream;
      stream.setRawMode(true);
      stream.on("data", (data: Buffer) => {
        if (!drainedEarly) this.earlyInputBuffer.push(Buffer.from(data));
        this.dataCallback?.(data);
      });
    },
    on(_event, handler) { this.dataCallback = handler; },
    pause() {
      if (this.stdin && "setRawMode" in this.stdin) {
        (this.stdin as tty.ReadStream).setRawMode(false);
      }
      this.stdin?.removeAllListeners("data");
      this.stdin?.destroy();
      this.stdin = null;
    },
    resume() { this.init(); },
    dispose() {
      if (this.stdin && "setRawMode" in this.stdin) {
        (this.stdin as tty.ReadStream).setRawMode(false);
      }
      this.stdin?.removeAllListeners("data");
      this.stdin?.destroy();
      this.stdin = null;
      this.dataCallback = null;
      this.earlyInputBuffer = [];
    },
    getEarlyInputText() {
      drainedEarly = true;
      if (this.earlyInputBuffer.length === 0) return "";
      const text = Buffer.concat(this.earlyInputBuffer).toString("utf8");
      this.earlyInputBuffer = [];
      return stripAnsiEscapes(text);
    },
  };
  source.init();
  return source;
}
```

### resolveCliContext Fix

```typescript
// Source: amp's S8() in cli-entrypoint.js:922-923
export function resolveCliContext(program: Command): CliContext {
  const opts = program.opts();
  
  // CRITICAL: Only check stdout for execute mode, not stderr
  const executeMode = Boolean(opts.execute) || 
    (!process.stdout.isTTY && !Boolean(opts.streamJson));
  
  // isTTY = both stdout AND stderr are TTY (for formatting decisions)
  const isTTY = Boolean(process.stdout.isTTY && process.stderr.isTTY);

  return {
    executeMode,
    isTTY,
    headless: Boolean(opts.headless),
    streamJson: Boolean(opts.streamJson) || Boolean(opts.headless),
    verbose: Boolean(opts.verbose),
    userMessage: program.args.length > 0 ? program.args.join(" ") : undefined,
  };
}
```

### ThreadStateWidget Reactive Subscription

```typescript
// Source: amp's ID0/k40 pattern in cli-commander-system.js:5966-6036
export class ThreadStateWidgetState extends State<ThreadStateWidget> {
  private _subscription: Subscription | null = null;
  private _eventSub: Subscription | null = null;
  private _messages: Message[] = [];
  private _inferenceState: InferenceState = "idle";
  private _error: Error | null = null;

  initState(): void {
    super.initState();
    const { threadStore, threadWorker, threadId } = this.widget.config;
    
    // Subscribe to thread snapshot changes
    const thread$ = threadStore.observeThread(threadId);
    if (thread$) {
      this._subscription = thread$.subscribe((snapshot) => {
        this.setState(() => {
          this._messages = snapshot.messages;
        });
      });
    }
    
    // Subscribe to worker events for streaming/errors
    this._eventSub = threadWorker.events$.subscribe((event) => {
      switch (event.type) {
        case "inference:start":
          this.setState(() => { this._inferenceState = "running"; });
          break;
        case "inference:error":
          this.setState(() => { this._error = event.error; this._inferenceState = "idle"; });
          break;
        case "turn:complete":
          this.setState(() => { this._inferenceState = "idle"; this._error = null; });
          break;
      }
    });
  }

  dispose(): void {
    this._subscription?.unsubscribe();
    this._eventSub?.unsubscribe();
    super.dispose();
  }
}
```

   - What we know: Each message needs markdown rendering via micromark pipeline
   - What's unclear: Whether to render all messages on every rebuild or cache rendered widgets
   - Recommendation: Start with full re-render on each build; optimize with caching if performance is poor (the frame scheduler already handles 16ms frame budget)

   - What's unclear: Whether flitter needs this -- if executeMode routing is fixed, stdout should always be TTY in interactive mode
