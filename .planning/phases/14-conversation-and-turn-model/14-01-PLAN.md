Per 14-CONTEXT.md: "Group thinking + text + tool calls into a single AssistantTurn (matches Amp's visual grouping)."

```typescript
/**
 * An assistant turn groups all items from a single assistant response.
 *
 * Contains zero or more thinking blocks, zero or one assistant text message,
 * zero or more tool calls, and zero or more plan items. The grouping boundary
 * is the next UserMessage or end of items array.
 */
export interface AssistantTurn {
  kind: 'assistant';
  /** Thinking blocks in this turn (may be empty). */
  thinkingItems: ReadonlyArray<ThinkingItem>;
  /** The assistant text message (null if response is tool-only). */
  message: AssistantMessage | null;
  /** Tool calls in this turn (may be empty). */
  toolCalls: ReadonlyArray<ToolCallItem>;
  /** Plan items in this turn (may be empty). */
  planItems: ReadonlyArray<PlanItem>;
  /** System messages in this turn (may be empty). */
  systemMessages: ReadonlyArray<SystemMessage>;

  // --- Computed turn-level properties ---

  /** True if any child item is currently streaming (thinking, text, or tool). */
  isStreaming: boolean;
  /** True if all children are finalized and no streaming is active. */
  isComplete: boolean;
  /** True if this turn was interrupted (user cancelled mid-response). */
  isInterrupted: boolean;
  /** True if this turn contains at least one tool call. */
  hasToolCalls: boolean;
  /** True if this turn contains at least one thinking block. */
  hasThinking: boolean;
  /** Turn status derived from children states. */
  status: TurnStatus;

  /** Start index in SessionState.items for the first item in this turn. */
  startIndex: number;
  /** End index (exclusive) in SessionState.items for items in this turn. */
  endIndex: number;
}
```

Create `packages/flitter-cli/src/state/conversation.ts` implementing `ConversationState` — a read-only view over `SessionState.items` that provides turn-level grouped access.

ConversationState design:
- Constructor takes a `SessionState` reference (composition, not inheritance)
- Internally caches the computed turns array and the SessionState version it was computed from
- `get turns(): ReadonlyArray<Turn>` — lazily recomputes when SessionState.version changes, otherwise returns cached
- `get currentTurn(): Turn | null` — last turn in the array
- `get turnCount(): number` — number of turns
- `get isEmpty(): boolean` — true when items array is empty
- `get lastUserTurn(): UserTurn | null` — last UserTurn in the array
- `get lastAssistantTurn(): AssistantTurn | null` — last AssistantTurn in the array

The core grouping algorithm (`groupItemsIntoTurns`):
1. Iterate through `SessionState.items` linearly
2. When a `user_message` is encountered:
   - If there's a pending assistant turn being accumulated, finalize it and push to results
   - Create a new `UserTurn` and push to results
   - Start accumulating a new assistant turn
3. For any non-user-message item:
   - If no assistant turn is being accumulated yet (items before first user message), start one
   - Route to the appropriate AssistantTurn bucket: thinking -> thinkingItems, assistant_message -> message, tool_call -> toolCalls, plan -> planItems, system_message -> systemMessages
4. After iteration, finalize any pending assistant turn
5. Compute AssistantTurn derived properties:
   - `isStreaming`: any thinking.isStreaming || message?.isStreaming || any toolCall.isStreaming
   - `isComplete`: !isStreaming && (message !== null || toolCalls.length > 0 || thinkingItems.length > 0)
   - `isInterrupted`: check if the preceding UserMessage has `interrupted === true`
   - `hasToolCalls`: toolCalls.length > 0
   - `hasThinking`: thinkingItems.length > 0
   - `status`: isStreaming ? 'streaming' : isInterrupted ? 'interrupted' : (all tools failed ? 'error' : 'complete')

Export the `groupItemsIntoTurns` function as a named export for direct testability.
