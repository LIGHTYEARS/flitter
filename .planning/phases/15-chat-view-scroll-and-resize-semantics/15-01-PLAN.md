## Implementation Steps

### Step 1: Create `ChatView` StatefulWidget

**File:** `packages/flitter-cli/src/widgets/chat-view.ts`

ChatView is a StatefulWidget that:
- Takes `AppState` as a constructor parameter
- Registers an AppState listener in `initState()`, removes it in `dispose()`
- Calls `setState()` on AppState change to trigger rebuild
- In `build()`, reads `appState.screenState` and dispatches to the appropriate sub-builder

```
ChatView (StatefulWidget)
  └── ChatViewState (State)
        ├── initState(): appState.addListener(onChange)
        ├── dispose(): appState.removeListener(onChange)
        ├── onChange(): this.setState()
        └── build():
              switch (screenState.kind) {
                'welcome'    → buildWelcomeScreen()
                'empty'      → buildEmptyScreen()
                'loading'    → buildLoadingScreen()
                'processing' → buildConversationView()   // same as ready, streaming shows cursor
                'error'      → buildErrorScreen(error)
                'ready'      → buildConversationView()
              }
```

**Key decisions:**
- ChatView owns the AppState listener lifecycle (not the parent)
- ScreenState dispatch happens at the top level of build()
- `processing` and `ready` both render the conversation — the difference is in turn-level streaming indicators, which are already encoded in `AssistantTurn.isStreaming`

### Step 2: Implement Turn Rendering

**File:** `packages/flitter-cli/src/widgets/chat-view.ts` (same file)

The conversation view builds a Column of turn widgets:

```
Column (mainAxisSize: 'min', crossAxisAlignment: 'stretch')
  ├── SizedBox(height: 1)              // spacer before first turn (optional)
  ├── UserTurnWidget(turn)             // for kind='user'
  ├── SizedBox(height: 1)              // inter-turn spacer
  ├── AssistantTurnWidget(turn)        // for kind='assistant'  → StickyHeader
  ├── SizedBox(height: 1)
  ├── UserTurnWidget(turn)
  ├── SizedBox(height: 1)
  └── AssistantTurnWidget(turn)        // last turn, possibly streaming
```

**UserTurnWidget** — renders the user message with Amp-style left border:
- Left border: green (normal), yellow (interrupted), bright cyan (selected)
- Text: italic, colored to match border
- Container with BoxDecoration (left BorderSide, width 2)
- Image attachment count if present

**AssistantTurnWidget** — wraps children in a StickyHeader:
- Header: empty SizedBox.shrink() (matches Amp — no "Claude" label)
- Body: Column of child widgets rendered from AssistantTurn fields
  - ThinkingItem placeholders (text: "[thinking...]", dim)
  - AssistantMessage text (plain text for now — markdown rendering deferred to Phase 19)
  - ToolCall placeholders (text: "[tool: {title}]", dim — full rendering deferred to Phase 18)
  - PlanItem placeholders (deferred to Phase 19)
  - SystemMessage separators (horizontal rule + dim italic text)

**Placeholder strategy:** Phase 15 renders structural placeholders for content types that have dedicated rendering phases later (tools, markdown, thinking, plans). This ensures the layout, scroll, and resize behavior is correct now, and specialized renderers drop in later without changing the layout contract.

### Step 3: Implement Screen Placeholders

**File:** `packages/flitter-cli/src/widgets/chat-view.ts` (same file)

Each non-conversation screen state gets a simple placeholder widget:

These are minimal but structurally correct — Phase 20 (Status, Themes, Motion) will add the full visual treatment.

### Step 4: Create App Shell Widget

**File:** `packages/flitter-cli/src/widgets/app-shell.ts`

Replace `bootstrap-shell.ts` as the root widget tree with the full app layout:

```
AppShell (StatefulWidget)
  └── FocusScope (autofocus, onKey: global key handler)
        └── Column (mainAxisSize: 'max', crossAxisAlignment: 'stretch')
              ├── Expanded
              │   └── Row (crossAxisAlignment: 'stretch')
              │       ├── Expanded
              │       │   └── [scroll infrastructure — wired in Plan 02]
              │       │       └── Padding (left: 2, right: 2, bottom: 1)
              │       │           └── ChatView (appState)
              │       └── [scrollbar — wired in Plan 02]
              ├── [InputArea placeholder — Phase 16]
              └── [StatusBar placeholder — Phase 20]
```

For Plan 01, the scroll infrastructure is not yet wired (that is Plan 02). The Expanded area directly contains `Padding > ChatView` so the widget tree is renderable and testable. Plan 02 wraps it in `SingleChildScrollView` + `Scrollbar`.

**Key handler (minimal for Plan 01):**
- `Ctrl+C`: cancel prompt if processing, else exit
- `Esc`: exit
- Other keys: ignored (Phase 16 adds full input handling)

## Verification

### Unit tests (in Plan 03)

1. ChatView.build() returns Center for welcome/empty/loading/error screen states
2. ChatView.build() returns Column with turn widgets for ready/processing states
3. UserTurnWidget renders left-bordered container with correct colors (normal, interrupted, selected)
4. AssistantTurnWidget wraps children in StickyHeader
5. Inter-turn SizedBox(height: 1) spacers are present between turn groups
6. ChatView listens to AppState and rebuilds on state change
7. ChatView disposes its listener on unmount

### Structural checks

- The conversation Column uses `mainAxisSize: 'min'` (critical for scroll — content-sized, not viewport-sized)
- The conversation Column uses `crossAxisAlignment: 'stretch'` (fills width)
- Turn count matches AppState.turns.length
- Empty turns array with welcome screenState renders welcome, not an empty Column

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Placeholder renderers make the chat look incomplete | Acceptable — each deferred phase (18, 19, 20) replaces its placeholder. The structural layout is correct now. |
| StatefulWidget listener leak | dispose() removes the listener. Test verifies this. |
| ChatView rebuild frequency during streaming | AppState notifies on every stream chunk. This is acceptable for Plan 01. Performance optimization (if needed) is deferred. |
