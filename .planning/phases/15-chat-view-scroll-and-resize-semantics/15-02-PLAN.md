## Implementation Steps

### Step 1: Create ScrollController in AppShellState

**File:** `packages/flitter-cli/src/widgets/app-shell.ts` (modify)

AppShellState creates and owns a ScrollController:

```typescript
class AppShellState extends State<AppShell> {
  private scrollController = new ScrollController();

  dispose(): void {
    this.scrollController.dispose();
    super.dispose();
  }
}
```

The ScrollController is shared between SingleChildScrollView and Scrollbar via the same instance reference.

### Step 2: Wrap ChatView in SingleChildScrollView

**File:** `packages/flitter-cli/src/widgets/app-shell.ts` (modify)

Replace the direct `Padding > ChatView` with the full scroll stack:

```
Expanded
  └── screenState needs scroll?
      YES (ready/processing) →
        Row (crossAxisAlignment: 'stretch')
          ├── Expanded
          │   └── SingleChildScrollView
          │         controller: this.scrollController
          │         position: 'bottom'
          │         enableKeyboardScroll: true
          │         enableMouseScroll: true

      NO (welcome/empty/loading/error) →
        Center
          └── ChatView (appState)   // screen placeholders center themselves
```

**Key layout decisions (Amp parity from reference app.ts):**

1. **Welcome/empty/loading/error bypass ScrollView** — these screens use `Center` for vertical centering. ScrollView gives unbounded height, which prevents `mainAxisAlignment: 'center'` from working. This matches the Amp BUG-1 FIX pattern observed in `flitter-amp/src/app.ts:1020`.

2. **Conversation screens use ScrollView + Scrollbar in a Row** — exact same layout as `flitter-amp/src/app.ts:1036-1066`.

3. **Scrollbar is always present in conversation mode** — 1 column wide, right-aligned, proportional thumb.

### Step 3: Follow Mode Semantics

Follow mode is already built into `ScrollController` and `RenderScrollViewport`. The behavior:

1. **Default: follow mode ON** — `ScrollController` starts with `_followMode = true`
2. **New content auto-scrolls** — `updateMaxScrollExtent()` auto-jumps to new max when `followMode && wasAtBottom`
3. **Manual scroll up disables follow** — `Scrollable._handleKey` calls `ctrl.disableFollowMode()` before scrolling up (ArrowUp, PageUp, Ctrl+U, 'k', 'g')
4. **Manual scroll down re-enables at bottom** — `jumpTo()` re-enables follow when `atBottom` detected
5. **Mouse scroll up disables follow** — `Scrollable._handleScroll` calls `disableFollowMode()` on wheel-up

**No additional code needed** — this behavior is built into flitter-core's ScrollController and Scrollable. Verify in Plan 03 tests.

### Step 4: Streaming Growth Stability (CHAT-03)

During streaming, the content grows as new text chunks arrive:

1. AppState listener fires → ChatView setState → rebuild → Column grows
2. RenderScrollViewport.performLayout() recalculates child size
3. ScrollController.updateMaxScrollExtent() called with new extent
4. If followMode is active, offset auto-jumps to new max → user sees latest content
5. If followMode is off (user scrolled up), offset stays put → content grows below viewport, user's view is stable

**This is already correct** — the interaction between `updateMaxScrollExtent()` and `followMode` in `ScrollController` handles this exact case. No custom code needed.

The only concern is rebuild frequency: during fast streaming, every chunk triggers a full rebuild. This is acceptable for correctness. If profiling reveals performance issues, the mitigation is batched flush scheduling (debounce rebuilds to ~16ms frames), which can be added later without changing the layout.

### Step 5: Resize Handling (CHAT-04)

Terminal resize triggers `WidgetsBinding.onResize()` → re-layout from root. The scroll stack handles this:

1. `RenderScrollViewport.performLayout()` receives new constraints from parent
2. Viewport size changes → `ScrollController.updateViewportSize()` called
3. Max scroll extent recalculated: `max(0, childSize - viewportSize)`
4. `ScrollController.updateMaxScrollExtent()` clamps offset if it exceeds new max
5. If followMode active, stays at bottom
6. If scrolled to a manual position, offset is clamped to valid range

**Content-relative scroll preservation:** When the viewport shrinks (terminal gets smaller), the current offset may exceed the new max. The controller clamps it. When the viewport grows (terminal gets bigger), more content is visible and the offset remains valid. This preserves usability — the user does not lose their position.

**No additional code needed** — flitter-core's layout pipeline and ScrollController handle resize through the standard constraint propagation path.

### Step 6: Scrollbar Appearance

The Scrollbar uses these settings (matching Amp):

```typescript
new Scrollbar({
  controller: this.scrollController,
  thumbColor: Color.brightBlack,
  trackColor: Color.defaultColor,  // dim track
  // defaults: thickness=1, subCharacterPrecision=true, interactive=true
})
```

- **Sub-character precision**: enabled by default — uses Unicode block elements for smooth thumb positioning
- **Interactive**: enabled by default — click-to-jump and thumb dragging
- **Thumb minimum extent**: 1 character (default)

Theme colors will be refined in Phase 20. For now, brightBlack thumb on default track matches the Amp visual.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Follow mode auto-scroll causes visual jitter during fast streaming | flitter-core's ScrollController updates synchronously in layout, not via scheduled callbacks. This should be smooth. Monitor during manual testing. |
| Scrollbar visual mismatch with Amp | Colors are placeholders — Phase 20 (themes) will provide the exact color tokens. Structural behavior is correct. |
| Center wrapping for non-conversation screens changes constraint chain vs Plan 01 | Plan 01 already documents this as the intended structure. ChatView screen placeholders are designed to work inside Center. |
