## Test Files

### File 1: `packages/flitter-cli/src/__tests__/chat-view.test.ts`

Tests the ChatView widget tree structure, screen state dispatch, and turn rendering.

#### Test Group 1: Screen State Dispatch

Tests that ChatView.build() returns the correct widget structure for each ScreenState variant.

```
1.1  welcome screenState → returns Center-based welcome screen
1.2  empty screenState → returns Center-based empty state with prompt hint
1.3  loading screenState → returns Center-based loading indicator
1.4  error screenState → returns Center-based error display with error message
1.5  ready screenState → returns Column-based conversation view
1.6  processing screenState → returns Column-based conversation view (same as ready)
```

**Test technique:** Create AppState with controlled session state, call `chatViewState.build()`, inspect the returned widget tree. Use helper functions to walk the widget tree and find expected widget types.

#### Test Group 2: Turn Rendering Structure

Tests the conversation Column layout for correctness.

```
2.1  Single UserTurn → Column with one UserTurn widget (left-bordered container)
2.2  User + Assistant pair → Column with UserTurnWidget, SizedBox(1), AssistantTurnWidget
2.3  Multiple turn pairs → correct alternation with SizedBox spacers between each
2.4  AssistantTurn wraps in StickyHeader with shrink header and Column body
2.5  Conversation Column uses mainAxisSize:'min' and crossAxisAlignment:'stretch'
2.6  Turn count in rendered Column matches AppState.turns.length (accounting for spacers)
```

#### Test Group 3: UserTurnWidget Rendering

Tests the visual structure of user message rendering.

```
3.1  Normal user message → green left border (width 2), italic green text
3.2  Interrupted message → yellow left border, yellow text
3.3  Selected message → bright cyan left border (width 3), bold text
3.4  Message with images → shows attachment count text
3.5  Container uses BoxDecoration with left BorderSide only
```

#### Test Group 4: AssistantTurnWidget Content

Tests that assistant turn children are rendered in the correct order.

```
4.1  Turn with only message → single text widget in StickyHeader body
4.2  Turn with thinking + message → thinking placeholder before message text
4.3  Turn with message + tool calls → message text then tool placeholders
4.4  Turn with system messages → horizontal rule separators rendered
4.5  Streaming turn → message text includes streaming indicator (cursor placeholder)
4.6  Empty assistant turn (no content) → StickyHeader with empty body
```

#### Test Group 5: AppState Listener Lifecycle

Tests that ChatView correctly manages its AppState listener.

```
5.1  initState registers listener on AppState
5.2  dispose removes listener from AppState
5.3  AppState change triggers setState (rebuild)
5.4  Multiple rapid state changes each trigger rebuild
```

**Test technique:** Create AppState, instantiate ChatViewState with mock lifecycle, verify listener count changes.

### File 2: `packages/flitter-cli/src/__tests__/app-shell-scroll.test.ts`

Tests the scroll infrastructure, follow mode, and resize behavior.

#### Test Group 6: Layout Structure

Tests the AppShell widget tree for correct scroll wiring.

```
6.1  Conversation mode: Expanded contains Row with Expanded(ScrollView) + Scrollbar
6.2  Welcome mode: Expanded contains Center (no ScrollView)
6.3  ScrollView uses position:'bottom' and enableKeyboardScroll:true
6.4  ScrollView and Scrollbar share the same ScrollController instance
6.5  Root Column uses mainAxisSize:'max' and crossAxisAlignment:'stretch'
```

#### Test Group 7: Follow Mode Behavior

Tests follow mode semantics using ScrollController directly (unit-level, no full render pipeline needed).

```
7.1  ScrollController starts with followMode=true
7.2  updateMaxScrollExtent auto-scrolls when followMode=true and atBottom
7.3  updateMaxScrollExtent does NOT auto-scroll when followMode=false
7.4  scrollBy(negative) via disableFollowMode + scrollBy does not auto-scroll on next extent update
7.5  jumpTo(maxExtent) re-enables followMode
7.6  Simulated streaming: repeated updateMaxScrollExtent with followMode=true tracks bottom
7.7  Simulated manual review: disableFollowMode, then updateMaxScrollExtent, offset stays put
```

**Note:** Tests 7.1-7.7 exercise ScrollController directly. These overlap with flitter-core's existing ScrollController tests but are included here to lock the specific behavioral contract that Chat View depends on. If flitter-core tests already cover these exact scenarios, these can be reduced to integration-level assertions.

#### Test Group 8: Keyboard Scroll Integration

Tests that keyboard scroll events correctly update the scroll controller.

```
8.1  'j' / ArrowDown scrolls down by 1
8.2  'k' / ArrowUp scrolls up by 1 and disables followMode
8.3  'g' jumps to top and disables followMode
8.4  'G' jumps to bottom
8.5  PageDown scrolls by viewport size
8.6  PageUp scrolls by viewport size and disables followMode
8.7  Ctrl+D scrolls by half viewport
8.8  Ctrl+U scrolls by half viewport and disables followMode
```

**Test technique:** Create ScrollableState, simulate KeyEvent dispatch, verify controller.offset and controller.followMode.

#### Test Group 9: Mouse Scroll Integration

Tests mouse wheel scroll behavior.

```
9.1  Mouse wheel up (button 64) scrolls up by mouseScrollStep and disables followMode
9.2  Mouse wheel down (button 65) scrolls down by mouseScrollStep
9.3  Mouse wheel at bottom re-enables followMode (via jumpTo/atBottom)
```

#### Test Group 10: Resize Behavior

Tests that resize (constraint change) preserves scroll usability.

```
10.1  Resize while followMode=true at bottom: remains at bottom after relayout
10.2  Resize while followMode=false at middle: offset clamped to valid range
10.3  Viewport shrink increases maxScrollExtent: offset stays valid
10.4  Viewport grow decreases maxScrollExtent: offset clamped if it exceeds new max
10.5  Content shorter than viewport after resize: no scrollbar needed (maxExtent=0)
```

**Test technique:** Create RenderScrollViewport, layout with initial constraints, update constraints (simulate resize), verify controller state.

#### Test Group 11: Scrollbar-ScrollView Coordination

Tests that the scrollbar correctly reflects scroll state.

```
11.1  Scrollbar with controller renders without error
11.2  Scrollbar reflects controller offset changes
11.3  Scrollbar hidden when content fits viewport (maxExtent=0)
```

## Verification Strategy

### Render tree assertions (per AGENTS.md testing rules)

Tests verify the widget tree structure, not just data state. Every test that checks "renders correctly" inspects the actual Widget returned by `build()`, not just the AppState/ConversationState data.

### Removal/cleanup path testing (per AGENTS.md testing rules)

Test 5.2 verifies that `dispose()` removes the AppState listener. Test 7.4 verifies that follow mode stays disabled after manual scroll.

### Screen buffer assertions where applicable

For scroll position tests (Groups 7-10), we verify controller.offset directly — this is the value that drives pixel-level rendering in RenderScrollViewport.paint(). Full screen buffer assertions (actual painted output verification) can be added in Phase 22 (Test Matrix) for end-to-end validation.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Widget tree inspection is brittle if internal structure changes | Tests inspect stable public API (widget type, key properties) not internal nesting depth |
| Scroll tests duplicate flitter-core coverage | Acceptable — these tests lock the specific behavioral contract for chat scroll. Overlap is intentional. |
| Test count is high (~58) | Each test is small and fast. The pattern established in Phases 13-14 shows this is manageable. |
