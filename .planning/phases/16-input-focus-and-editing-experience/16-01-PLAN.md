**Phase:** 16 — Input, Focus, and Editing Experience

## Goal

Create the `InputArea` StatefulWidget for `flitter-cli` that provides Amp-parity multi-line text editing with border overlays, mode badges, shell-mode detection, auto-expanding height, and submit behavior.

## Context

- `flitter-core` already provides `TextField`, `TextEditingController`, `Container`, `BoxDecoration`, `Border`, `BorderSide`, `Stack`, `Positioned`, `ContainerWithOverlays`, and `Autocomplete` — all needed primitives exist.
- `flitter-amp/src/widgets/input-area.ts` is the reference implementation (432 lines). We port the design, not copy verbatim, adapting to `flitter-cli`'s native state model.
- The current `AppShell` has a `// InputArea placeholder -- Phase 16` comment at line 216 where InputArea will be inserted (Plan 03 handles wiring).
- `AppState` already exposes `submitPrompt(text)`, `cancelPrompt()`, `isProcessing`, `currentMode`, and the listener pattern.

## Implementation Steps

### Step 1: Create `packages/flitter-cli/src/widgets/input-area.ts`

Create a new `InputArea` StatefulWidget with the following structure:

```
InputArea (StatefulWidget)
  Props:
    - onSubmit: (text: string) => void
    - isProcessing: boolean
    - mode: string | null           // agent mode label ("smart", "code", etc.)
    - controller?: TextEditingController  // external controller (optional)
    - maxExpandLines?: number        // default 10
    - overlayTexts?: BorderOverlayText[] // Phase 17+ extension point

  State:
    - controller: TextEditingController (owned or external)
    - currentText: string            // cached for shell-mode diff detection
    - dragHeight: number | null      // manual resize override
```

### Step 2: Shell Mode Detection

Port the `detectShellMode()` function from `flitter-amp/src/widgets/input-area.ts`:

```typescript
type ShellMode = 'shell' | 'background' | null;

/** Detect shell mode from input text prefix. */
function detectShellMode(text: string): ShellMode {
  if (text.startsWith('$$')) return 'background';
  if (text.startsWith('$'))  return 'shell';
  return null;
}
```

Shell mode changes the border color (cyan for shell mode) and the mode badge label.

### Step 3: Auto-Expanding Height

Port the height computation logic:

- `MIN_HEIGHT = 3` (1 content line + 2 border rows)
- Auto-expand: `height = min(lineCount, maxExpandLines) + 2`
- Manual drag override: `dragHeight` takes precedence when set
- Clear `dragHeight` on submit (input cleared -> reset to auto)

### Step 4: Build Method — Widget Tree

```
Stack (fit: 'passthrough')
  Container (bordered, padded, height computed)
    TextField
      - controller: this.controller
      - autofocus: true (will get focus via FocusScope — Plan 02)
      - submitOnEnter: true (Enter submits)
      - cursorChar: '\u2588' (block cursor)
      - onSubmit: _handleSubmit
  Positioned (top:0, right:1) — mode badge (optional)
  Positioned (top:0, left:0, right:0) — drag resize MouseRegion (height:1)
```

Key build details:
- Border: `Border.all(new BorderSide({ color: borderColor, width: 1, style: 'rounded' }))`
- Border color: `shellMode ? Color.cyan : Color.brightBlack`
- Padding inside border: `EdgeInsets.symmetric({ horizontal: 1 })`
- Mode badge: shows shell label or agent mode, with processing spinner when `isProcessing`

### Step 5: Submit Handler

```typescript
private _handleSubmit = (text: string): void => {
  if (text.trim().length > 0 && !this.widget.isProcessing) {
    this.widget.onSubmit(text.trim());
    this.controller.clear();
    this.dragHeight = null;  // reset manual resize
  }
};
```

Guards:
- Empty text: no-op
- Processing: no-op (double-submit prevention)

### Step 6: Text Change Handler

Listen to controller changes to detect:
- Shell mode transitions (border color change)
- Line count changes (auto-expand height change)

Only call `setState()` when visual properties actually change (shell mode or line count), not on every keystroke, to avoid unnecessary rebuilds.

### Step 7: Mode Badge Overlay

Render the mode/shell badge at `Positioned({ top: 0, right: 1 })`:

- Shell mode labels: `"Shell mode"` or `"Background shell"`
- Agent mode labels: passed via `this.widget.mode` (e.g., `"smart"`, `"code"`)
- Processing dimming: when `isProcessing && !shellLabel`, dim the label
- Color: shell labels use `Color.cyan`, agent mode uses `Color.green` (placeholder — theme-aware color in Phase 20)

### Step 8: Drag Resize Support

Port the top-border mouse region for drag-resize:
- `MouseRegion` at `Positioned({ top: 0, left: 0, right: 0 })` with `height: 1`
- Press: record `dragStartY` and current computed height
- Drag: compute delta, clamp to `[MIN_HEIGHT, floor(screenHeight/2)]`
- Release: clear drag state

### Step 9: Autocomplete Stub

Wire `Autocomplete` around the `TextField` with empty triggers. This establishes the integration point for Phase 17 (file picker, `@` trigger):

```typescript
const autocompleteWrapped = new Autocomplete({
  child: textField,
  controller: this.controller,
  triggers: [],  // Phase 17 populates these
});
```

### Step 10: Export BorderOverlayText Type

Export the `BorderOverlayText` interface for downstream consumers:

```typescript
export interface BorderOverlayText {
  position: 'top-left' | 'top-right' | 'bottom-left' | 'bottom-right';
  child: Widget;
}
```

## Decisions

- Use `Stack + Positioned` directly (not `ContainerWithOverlays`) for the overlay pattern, matching the flitter-amp reference which also uses raw Stack/Positioned for full control over overlay placement.
- `submitOnEnter: true` on TextField means Enter submits, Shift+Enter inserts newline. This matches the Amp input behavior (single-line submit semantic with multi-line support via Shift+Enter).
- No theme integration in this plan. Colors use hardcoded defaults (brightBlack border, cyan shell, green mode). Phase 20 introduces the theme system.
- No external editor integration in this plan. `Ctrl+G` (external editor) is wired in Plan 02 as a stub that logs a warning.
