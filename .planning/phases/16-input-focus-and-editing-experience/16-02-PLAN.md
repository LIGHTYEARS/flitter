# Plan 16-02: Focus Routing, Global Shortcuts, and Shell Mode Actions

**Phase:** 16 — Input, Focus, and Editing Experience
**Requirements:** INPT-03, INPT-05, INPT-06
**Depends on:** Plan 16-01 (InputArea widget exists)
**Status:** Pending
**Estimated new tests:** 0 (behavioral wiring; tests in Plan 04)

## Goal

Wire the focus routing system so InputArea has default focus, overlays capture focus when active, and key events bubble correctly. Implement the complete global shortcut matrix matching Amp semantics. Add the external editor stub for `Ctrl+G`.

## Context

- flitter-core's `FocusManager.dispatchKeyEvent()` bubbles key events from the focused node up through ancestors until one returns `'handled'`.
- The current `AppShell` has a `FocusScope({ autofocus: true, onKey: _handleKey })` at root level that handles `Ctrl+C` and `Esc`. This needs to expand to the full shortcut matrix.
- InputArea's `TextField` will have its own `FocusScope({ autofocus: true })`, making it the default focus target. Key events the TextField does not handle bubble up to AppShell's FocusScope.
- Overlay focus capture is Phase 17's concern (permission dialog, command palette). This plan establishes the focus return pattern that Phase 17 will use.

## Focus Architecture

```
FocusScope (AppShell — global key handler)
  └── Column
      ├── Expanded (ChatView — no focus node, transparent)
      ├── InputArea
      │   └── FocusScope (TextField — autofocus, handles text input keys)
      └── [StatusBar — Phase 20, no focus]
```

Key event flow:
1. Terminal key event -> `FocusManager.dispatchKeyEvent(event)`
2. primaryFocus = TextField's FocusNode (has autofocus)
3. TextField handles printable chars, arrows, Backspace, Enter, Shift+Enter, etc. -> `'handled'`
4. Non-text keys (Ctrl+C, Ctrl+L, etc.) -> TextField returns `'ignored'`
5. Event bubbles to AppShell's FocusScope onKey handler
6. AppShell handles global shortcuts -> `'handled'` or `'ignored'`

## Implementation Steps

### Step 1: Expand AppShellState._handleKey() — Full Shortcut Matrix

Replace the minimal Ctrl+C/Esc handler with the complete matrix:

```typescript
private _handleKey(event: KeyEvent): KeyEventResult {
  // --- Global shortcuts (handled at AppShell level) ---

  if (event.ctrlKey && !event.shiftKey && !event.altKey) {
    switch (event.key.toLowerCase()) {
      case 'c':
        return this._handleCtrlC();
      case 'l':
        return this._handleCtrlL();
      case 'o':
        return this._handleCtrlO();
      case 'g':
        return this._handleCtrlG();
      case 'r':
        return this._handleCtrlR();
    }
  }

  if (event.altKey && !event.ctrlKey && !event.shiftKey) {
    switch (event.key.toLowerCase()) {
      case 't':
        return this._handleAltT();
    }
  }

  if (event.key === 'Escape') {
    return this._handleEscape();
  }

  // '?' when input is empty — show shortcut help
  if (event.key === '?' && !event.ctrlKey && !event.altKey && !event.shiftKey) {
    return this._handleQuestionMark();
  }

  return 'ignored';
}
```

private _handleQuestionMark(): KeyEventResult {

Note: The `?` shortcut has a design tension — TextField consumes all printable characters. The Amp solution is to check if the input is empty before the TextField handles it. This requires the InputArea to intercept `?` when the controller text is empty and return `'ignored'` to bubble it up. This interception is wired in Step 3.

### Step 3: InputArea Key Interception for `?` Empty-Input Shortcut

Add a `FocusScope` wrapper around the InputArea content that intercepts specific keys before the TextField gets them:

In `InputArea.build()`, wrap the Autocomplete-wrapped TextField:

```typescript
// The '?' key should bubble to AppShell when input is empty
// to trigger shortcut help. TextField would normally consume it.
const inputFocusScope = new FocusScope({
  onKey: (event: KeyEvent): KeyEventResult => {
    if (event.key === '?' && this.controller.text.length === 0) {
      return 'ignored';  // let it bubble to AppShell
    }
    return 'ignored';  // all other keys pass through to TextField
  },
  child: autocompleteWrapped,
});
```

Wait — this does not work because the FocusScope wrapping would intercept BEFORE the child gets the event if it returns `'handled'`, but if it returns `'ignored'` the child still gets it. The issue is that FocusScope's onKey runs on bubble-UP from the child, not before the child.

Correction: Key events dispatch to the **primaryFocus** (deepest focused node) first, then bubble UP through parents. So:

1. TextField gets `?` -> inserts character -> returns `'handled'`
2. Event never reaches AppShell

To implement the Amp behavior, the TextField itself needs to not handle `?` when empty. This requires either:
- (a) A custom `onKey` on the TextField that checks emptiness, or
- (b) A wrapper FocusNode that intercepts BEFORE the TextField's node

Option (b) is not possible with the current bubble-up architecture.
Option (a): The TextField's `submitOnEnter` + custom behavior. The cleanest approach is to add an `onBeforeInput` callback to InputArea that can veto character insertion.

**Decision:** Defer the `?` empty-input shortcut to Phase 17 (shortcut help overlay), where the command palette can implement it properly. The `?` handler in AppShell remains as a stub that fires only in non-TextField focus states.

### Step 4: TextEditingController Sharing

The AppShell needs access to the InputArea's controller for features like:
- Ctrl+G (external editor): read/write controller text
- Future: Tab completion, history reinjection

Add a `textController` property to AppShellState that is passed to InputArea:

```typescript
class AppShellState extends State<AppShell> {
  private scrollController = new ScrollController();
  private textController = new TextEditingController();

  // ... in build():
  new InputArea({
    onSubmit: (text) => this.widget.appState.submitPrompt(text),
    isProcessing: this.widget.appState.isProcessing,
    mode: this.widget.appState.currentMode,
    controller: this.textController,
  }),
```

This allows AppShellState to read `this.textController.text` for Ctrl+G and other shortcuts that need input context.

### Step 5: AppShellState Listener for InputArea Reactivity

The InputArea needs to re-render when `isProcessing` or `currentMode` changes. Since AppShell's `build()` reads these from `appState`, AppShellState needs an AppState listener that triggers `setState()`:

```typescript
class AppShellState extends State<AppShell> {
  private _onStateChange: (() => void) | null = null;

  initState(): void {
    super.initState();
    this._onStateChange = () => this.setState();
    this.widget.appState.addListener(this._onStateChange);
  }

  dispose(): void {
    if (this._onStateChange) {
      this.widget.appState.removeListener(this._onStateChange);
    }
    this.scrollController.dispose();
    this.textController.dispose();
    super.dispose();
  }
}
```

Note: ChatView already has its own AppState listener. The AppShell listener is needed because AppShell's build() now reads `isProcessing`, `currentMode`, and `screenState` to configure InputArea and the scroll/center decision. Without this listener, InputArea would show stale `isProcessing` state.

## Decisions

- `?` empty-input shortcut deferred to Phase 17. TextField consumes all printable characters, so implementing this requires either modifying TextField's key handling or adding a pre-dispatch interceptor. Both are too invasive for this plan.
- External editor (Ctrl+G / INPT-06) is a stub in this plan. Full implementation requires terminal suspend/restore which is a larger concern than shortcut wiring.
- History search (Ctrl+R) is a stub. Phase 21 dependency.
- Command palette (Ctrl+O) is a stub. Phase 17 dependency.
- AppShellState now owns the lifecycle of both ScrollController and TextEditingController. Both are disposed in `dispose()`.
- AppShellState registers an AppState listener because its `build()` method now reads dynamic properties (`isProcessing`, `currentMode`, `screenState`) to configure InputArea. Without this, the InputArea would render with stale props after state changes.
