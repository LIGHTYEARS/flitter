### Step 2: Add TextEditingController and AppState Listener to AppShellState

These were designed in Plan 02 but are physically added here since Plan 03 is the integration step:

```typescript
class AppShellState extends State<AppShell> {
  private scrollController = new ScrollController();
  private textController = new TextEditingController();
  private _onStateChange: (() => void) | null = null;

  initState(): void {
    super.initState();
    this._onStateChange = () => this.setState();
    this.widget.appState.addListener(this._onStateChange);
  }

  dispose(): void {
    if (this._onStateChange) {
      this.widget.appState.removeListener(this._onStateChange);
    }
    this.scrollController.dispose();
    this.textController.dispose();
    super.dispose();
  }
```

### Step 4: Verify Focus Flow

After integration, the focus tree will be:

```
FocusScope (AppShell, autofocus=true)
  └── FocusScope (TextField inside InputArea, autofocus=true)
```

Both have `autofocus: true`. The **deepest** autofocus node wins because `FocusScopeState.initState()` uses `queueMicrotask(() => requestFocus())`. The TextField's microtask runs after AppShell's, so TextField gets primaryFocus. This is the correct behavior — InputArea should have default focus.

Key bubble verification:
- Printable char -> TextField handles -> `'handled'` (never reaches AppShell)
- Ctrl+C -> TextField ignores (not in its key handler) -> bubbles to AppShell -> `'handled'`
- Ctrl+L -> TextField ignores -> bubbles to AppShell -> `'handled'`
- Enter -> TextField handles (submit) -> `'handled'` (never reaches AppShell)
- Shift+Enter -> TextField handles (newline) -> `'handled'`

### Step 5: Wire Submit Pipeline End-to-End

The complete submit flow:

1. User types text in TextField
2. User presses Enter
3. TextField calls `onSubmit(text)` -> InputArea's `_handleSubmit`
4. `_handleSubmit` validates (non-empty, not processing)
5. Calls `this.widget.onSubmit(text.trim())` -> AppShell's lambda
6. Lambda calls `this.widget.appState.submitPrompt(text)`
7. AppState delegates to PromptController
8. PromptController drives session: processing -> streaming -> complete
9. Session notifies listeners -> AppState notifies listeners
10. AppShellState listener triggers `setState()` -> rebuild
11. ChatView listener triggers `setState()` -> conversation updates
12. InputArea sees `isProcessing: true` -> submit guard active

### Step 6: Verify Cancel Pipeline

1. User presses Ctrl+C during processing
2. TextField ignores Ctrl+C -> bubbles to AppShell
3. AppShell checks `appState.isProcessing` -> true
4. Calls `appState.cancelPrompt()` -> PromptController.cancel()
5. Provider request aborted, session transitions to cancelled
6. Listeners fire -> rebuild -> `isProcessing: false`
7. InputArea re-enables submit

## Integration Risks

1. **Double-rebuild**: Both ChatView and AppShellState listen to AppState. This means each state change triggers TWO rebuilds — one from AppShellState's listener (rebuilds Column including InputArea), one from ChatView's listener (rebuilds ChatView internals). This is acceptable because:
   - The rebuild is cheap (widget creation, not layout/paint)
   - The framework's element reconciliation avoids redundant layout passes
   - This matches the pattern used in Phases 14-15

2. **Focus conflict**: Two `autofocus: true` FocusScopes. Verified that deepest wins (TextField), which is the desired behavior.

3. **ScrollController lifecycle**: ScrollController is created in AppShellState's field initializer. If AppShell is unmounted and remounted, a new ScrollController is created. This is correct because the scroll position should reset on remount.

## Decisions

- InputArea is a direct Column child (not wrapped in Expanded or Flexible). Its height is self-determined by the auto-expand computation. The Expanded wrapper around the chat area absorbs the remaining space.
- `onSubmit` lambda in `build()` closes over `this.widget.appState`. This is safe because `build()` is called on every state change, so the lambda always captures the current widget.
- No `key:` property on InputArea widget. Since InputArea is always present in the Column at the same index, element reconciliation will reuse the same element and update props.
