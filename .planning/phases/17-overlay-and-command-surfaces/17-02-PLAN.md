# Plan 17-02: Permission Dialog Overlay

**Phase:** 17 — Overlay and Command Surfaces
**Requirements:** OVLY-01
**Depends on:** Plan 17-01 (OverlayManager, overlay IDs, ShortcutRegistry)
**Status:** Pending
**Estimated new tests:** 0 (widget construction verified manually; automated tests in Phase 22)

## Goal

Implement the permission dialog overlay for `flitter-cli`. The dialog is a modal overlay with dimmed background, focus capture, approval options (allow / always-allow / deny), and Escape dismissal. It is shown when the provider/session layer requests permission for a tool call, and its result is resolved back to the caller.

## Context

- `flitter-amp/src/widgets/permission-dialog.ts` uses `Dialog` + `DialogOverlay` + `SelectionList` from `flitter-core`. These primitives already exist in `flitter-core/src/widgets/dialog.ts` and `flitter-core/src/widgets/dialog-overlay.ts`.
- The `flitter-amp` implementation receives a `RequestPermissionRequest` from the ACP SDK. In `flitter-cli`, there is no ACP SDK. The permission request must be defined as a native type.
- `OverlayManager` (from Plan 17-01) handles the modal mask rendering (semi-transparent background). The `PermissionDialog` widget itself renders inside a `DialogOverlay` which provides centered framing and border styling.
- The permission dialog is shown at priority `OVERLAY_PRIORITIES.PERMISSION_DIALOG` (100), the highest level, ensuring it always renders on top of other overlays.
- `AppState` needs a permission request/response flow: show the overlay, wait for user selection, return the result to the caller.

## Implementation Steps

### Step 1: Define native permission request types

Create `packages/flitter-cli/src/state/permission-types.ts`:

```typescript
/**
 * A permission request for a tool call that needs user approval.
 */
export interface PermissionRequest {
  /** Unique identifier for this request (for matching response). */
  readonly requestId: string;
  /** The tool call requiring approval. */
  readonly toolCall: {
    readonly title: string;
    readonly kind: string;
  };
  /** The available response options. */
  readonly options: readonly PermissionOption[];
}

/**
 * A single option in a permission dialog.
 */
export interface PermissionOption {
  readonly optionId: string;
  readonly name: string;
  /** Display description derived from the option kind. */
  readonly description: string;
}

/**
 * Result of a permission dialog interaction.
 * null means the user dismissed without selecting (Escape).
 */
export type PermissionResult = string | null;
```

### Step 2: Add permission flow to `AppState`

Modify `packages/flitter-cli/src/state/app-state.ts`:

Add a promise-based permission flow:

```typescript
/** Pending permission request resolver. */
private _permissionResolve: ((result: PermissionResult) => void) | null = null;

/**
 * Request permission from the user. Shows the permission dialog overlay
 * and returns a Promise that resolves when the user selects an option
 * or dismisses the dialog.
 */
async requestPermission(request: PermissionRequest): Promise<PermissionResult> {
  return new Promise<PermissionResult>((resolve) => {
    this._permissionResolve = resolve;
    this.overlayManager.show({
      id: OVERLAY_IDS.PERMISSION_DIALOG,
      priority: OVERLAY_PRIORITIES.PERMISSION_DIALOG,
      modal: true,
      placement: { type: 'fullscreen' },
      builder: (onDismiss) => new PermissionDialog({
        request,
        onSelect: (optionId: string) => {
          onDismiss();
          this.resolvePermission(optionId);
        },
        onCancel: () => {
          onDismiss();
          this.resolvePermission(null);
        },
      }),
    });
  });
}

/**
 * Resolve a pending permission request with the given result.
 * Called by the dialog's onSelect/onCancel or by Escape dismissal.
 */
resolvePermission(result: PermissionResult): void {
  if (this._permissionResolve) {
    const resolve = this._permissionResolve;
    this._permissionResolve = null;
    resolve(result);
  }
}
```

### Step 3: Create `PermissionDialog` widget

Create `packages/flitter-cli/src/widgets/permission-dialog.ts`:

Port the `flitter-amp/src/widgets/permission-dialog.ts` pattern but replace ACP SDK types with the native `PermissionRequest` from Step 1.

Widget structure:
```
DialogOverlay
  Dialog (title: 'Permission Required', type: 'warning')
    subtitle: "{toolCall.title} ({toolCall.kind})"
    body: SelectionList
      items: request.options mapped to SelectionItem[]
      onSelect: props.onSelect
      onCancel: props.onCancel
      showDescription: true
```

Key differences from flitter-amp version:
- No `AmpThemeProvider` lookup — use hardcoded Color values (brightYellow for warning border/title, white for subtitle). Phase 20 adds theme support.
- Import `PermissionRequest` from `../state/permission-types` instead of ACP SDK.

### Step 4: Wire Escape dismissal to resolve permission

In `packages/flitter-cli/src/shortcuts/defaults.ts`, update the `dismiss-overlay` shortcut action:

When the dismissed overlay ID is `OVERLAY_IDS.PERMISSION_DIALOG`, call `ctx.appState.resolvePermission(null)` to unblock the caller. This matches the `flitter-amp` pattern where Escape dismissal resolves the permission as denied.

```typescript
action: (ctx): KeyEventResult => {
  if (ctx.overlayManager.hasOverlays) {
    const dismissedId = ctx.overlayManager.dismissTop();
    if (dismissedId === OVERLAY_IDS.PERMISSION_DIALOG) {
      ctx.appState.resolvePermission(null);
    }
    return 'handled';
  }
  return 'ignored';
},
```

### Step 5: Integrate permission flow with PromptController

Modify `packages/flitter-cli/src/state/prompt-controller.ts`:

Add an optional `onPermissionRequest` callback that the provider can invoke when a tool call requires permission. The prompt controller calls `appState.requestPermission(request)` and awaits the result.

This is a forward-looking integration point. The actual tool permission flow in the provider layer will be wired when tool execution is implemented (Phase 18). For now, the plumbing exists so the permission dialog can be tested manually by calling `appState.requestPermission(...)` from a test harness.

## Verification

- [ ] `PermissionDialog` builds a `DialogOverlay` with title "Permission Required" and warning type
- [ ] `SelectionList` inside the dialog shows options from the `PermissionRequest`
- [ ] Selecting an option calls `onSelect(optionId)`, which dismisses the overlay and resolves the promise
- [ ] Pressing Escape dismisses the dialog and resolves the promise with `null`
- [ ] The dialog renders as modal (dimmed background via OverlayManager's modal mask)
- [ ] The dialog renders at priority 100, on top of any other overlays
- [ ] `appState.requestPermission(request)` returns a Promise that resolves to the selected optionId or null
- [ ] Focus is captured by the `FocusScope` inside `DialogOverlay` — keyboard input is isolated to the dialog

## Decisions

- **Promise-based permission flow.** `requestPermission()` returns a Promise, making it easy for the provider/tool layer to await user approval inline. This matches the Amp pattern where permission dialogs block tool execution.
- **No ACP SDK dependency.** `PermissionRequest` and `PermissionOption` are native flitter-cli types. When tool execution is implemented (Phase 18), the tool dispatch layer constructs these from its own tool metadata.
- **No theme integration yet.** Warning colors (brightYellow) are hardcoded. Phase 20 wires the theme system.
- **Forward-looking integration with PromptController.** The actual tool permission invocation from the provider is deferred to Phase 18. The dialog is fully functional through `appState.requestPermission()`.
