# Plan 17-05: Shortcut Help / Discovery Surface

**Phase:** 17 — Overlay and Command Surfaces
**Requirements:** OVLY-04, OVLY-05 (partial)
**Depends on:** Plan 17-01 (OverlayManager, overlay IDs, ShortcutRegistry)
**Status:** Pending
**Estimated new tests:** 0 (widget construction verified manually; automated tests in Phase 22)

## Goal

Implement the shortcut help/discovery overlay for `flitter-cli`. The overlay is invoked by pressing `?` when the input is empty (non-TextField focus states), displays all keyboard shortcuts grouped by category, and is the definitive discovery surface behind the status-area shortcut hints. The shortcut data is read from the `ShortcutRegistry` (Plan 17-01) so it cannot drift from actual bindings.

## Context

- `flitter-amp/src/widgets/shortcut-help-overlay.ts` is a 305-line `StatelessWidget` that displays grouped shortcut rows inside a bordered, centered overlay with a semi-transparent background mask. It optionally reads from a `ShortcutRegistry` and falls back to a hardcoded list.
- In `flitter-cli`, the `ShortcutRegistry` (Plan 17-01) is the single source of truth. There is no fallback list. All shortcuts visible in the overlay are derived from the registry.
- The overlay includes a static "Input" section showing widget-level behaviors (Enter to submit, `@` for file autocomplete, `$ ...` for shell mode) that are not registered shortcuts.
- The overlay is shown at priority `OVERLAY_PRIORITIES.SHORTCUT_HELP` (50), the same level as the command palette. Since they use the same priority, only one should be open at a time — the `?` shortcut guard (`enabled` predicate) already prevents opening when overlays are active.
- `?` toggles the overlay: pressing `?` while the overlay is open dismisses it.

## Implementation Steps

### Step 1: Create `ShortcutHelpOverlay` widget

Create `packages/flitter-cli/src/widgets/shortcut-help-overlay.ts`:

Port the `flitter-amp/src/widgets/shortcut-help-overlay.ts` pattern with these changes:
- **No fallback list.** Always derive groups from the `ShortcutRegistry`. The `registry` prop is required (not optional).
- Replace `AmpThemeProvider` color lookups with hardcoded defaults (cyan for info, white for foreground, brightBlack for muted, blue for keybind).
- Remove the internal `Stack` with modal mask. The `OverlayManager` handles the modal mask when `modal: true` is set on the entry.

Props:
```typescript
interface ShortcutHelpOverlayProps {
  onDismiss: () => void;
  registry: ShortcutRegistry;
}
```

Widget structure:

### Step 2: Derive groups from ShortcutRegistry

Create a `groupsFromRegistry()` helper inside the widget module:

```typescript
/**
 * Category display titles.
 */
const CATEGORY_TITLES: Record<ShortcutCategory, string> = {
  general: 'General',
  display: 'Display',
  navigation: 'Navigation',
  input: 'Input',
};

/**
 * Static input shortcuts not in the registry (widget-level behaviors).
 */
const INPUT_SHORTCUTS = [
  { key: 'Enter',    description: 'Submit prompt' },
  { key: '@',         description: 'Trigger file autocomplete' },
  { key: '$ ...',     description: 'Shell mode (run command)' },
  { key: '$$ ...',    description: 'Background shell mode' },
];

/**
 * Build display groups from the ShortcutRegistry.
 * Appends static Input shortcuts to the 'input' category.
 * Only includes non-empty groups.
 */
function groupsFromRegistry(registry: ShortcutRegistry): ShortcutGroup[] {
  const grouped = registry.getGroupedEntries();
  const result: ShortcutGroup[] = [];

  for (const [category, entries] of grouped) {
    const shortcuts = entries.map(e => ({
      key: e.displayKey,
      description: e.description,
    }));

    if (category === 'input') {
      shortcuts.push(...INPUT_SHORTCUTS);
    }

    if (shortcuts.length > 0) {
      result.push({
        title: CATEGORY_TITLES[category],
        shortcuts,
      });
    }
  }

  // Ensure Input section always appears even if no input-category shortcuts
  const hasInput = result.some(g => g.title === 'Input');
  if (!hasInput) {
    result.push({
      title: 'Input',
      shortcuts: INPUT_SHORTCUTS,
    });
  }

  return result;
}
```

### Step 3: Wire `?` shortcut to show/toggle overlay

In `packages/flitter-cli/src/shortcuts/defaults.ts` (Plan 17-01), the `toggle-shortcut-help` entry calls `ctx.hooks.showShortcutHelp()`.

In `packages/flitter-cli/src/widgets/app-shell.ts`, implement the `showShortcutHelp` hook:

```typescript
showShortcutHelp(): void {
  const overlayManager = this.widget.appState.overlayManager;
  if (overlayManager.has(OVERLAY_IDS.SHORTCUT_HELP)) {
    overlayManager.dismiss(OVERLAY_IDS.SHORTCUT_HELP);
    return;
  }
  overlayManager.show({
    id: OVERLAY_IDS.SHORTCUT_HELP,
    priority: OVERLAY_PRIORITIES.SHORTCUT_HELP,
    modal: true,
    placement: { type: 'fullscreen' },
    builder: (onDismiss) => new ShortcutHelpOverlay({
      onDismiss,
      registry: this.shortcutRegistry,
    }),
  });
}
```

The overlay is modal (`modal: true`) so the OverlayManager renders a semi-transparent background mask. This matches the Amp behavior where the shortcut help dims the background.

### Step 4: Handle key absorption in overlay

The `FocusScope.onKey` handler inside `ShortcutHelpOverlay` absorbs all keys:
- `Escape` -> dismiss
- `?` -> dismiss (toggle off)
- All other keys -> return `'handled'` (swallow, prevent propagation to the input)

This prevents typing while the help overlay is visible.

### Step 5: Ensure the overlay reads the same data as actual shortcuts

Because the overlay reads from `ShortcutRegistry.getGroupedEntries()`, any change to the default shortcuts in `defaults.ts` (Plan 17-01) automatically reflects in the help overlay. There is no separate list to keep in sync.

The static Input section (`Enter`, `@`, `$`, `$$`) is maintained as a constant because these are widget-level behaviors from TextField and InputArea, not registered shortcuts.

## Decisions

- **Registry as sole source of truth.** No fallback hardcoded list. If the registry is empty, the overlay shows only the static Input section. This eliminates the drift risk where shortcut help lists bindings that do not match actual behavior.
- **Modal overlay with key absorption.** The help overlay is informational and should prevent accidental input. All keys except Escape and `?` are swallowed.
- **Static Input section.** Enter, `@`, `$`, `$$` are widget-level behaviors (TextField/InputArea) that cannot be registered as global shortcuts. They are maintained as a static constant in the overlay module.
- **No theme integration yet.** Colors (cyan info, blue keybind, brightBlack muted, white foreground) are hardcoded. Phase 20 adds theme support.
- **Same priority as command palette (50).** The `?` shortcut's `enabled` guard prevents opening when any overlay is already active, so they cannot stack. If somehow both are open (e.g., programmatic), the most recently opened one sits on top.
