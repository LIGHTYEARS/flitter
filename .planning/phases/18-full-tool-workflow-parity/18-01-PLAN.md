# Plan 18-01: Tool Infrastructure — Dispatch, Name Resolution, Output Utilities

**Phase:** 18 — Full Tool Workflow Parity
**Requirements:** TOOL-01, TOOL-04 (partial), TOOL-05
**Depends on:** Phase 17 complete (AppShell, overlay stack, shortcut registry)
**Status:** Pending
**Estimated new tests:** 0 (infrastructure; tests deferred to Phase 22)

## Goal

Port the tool dispatch infrastructure from `flitter-amp` into `flitter-cli`: tool name resolution/normalization, base props contract, output extraction utilities, and truncation constants. This creates the foundation that all subsequent tool rendering plans depend on, without modifying any widgets yet.

## Context

- `flitter-amp/src/widgets/tool-call/resolve-tool-name.ts` provides `TOOL_NAME_MAP`, `resolveToolName()`, `resolveToolDisplayName()`, `shortenPath()`, and `extractTitleDetail()`. These are pure functions with no ACP SDK dependency — they only reference `ToolCallItem` (already ported to flitter-cli types.ts).
- `flitter-amp/src/widgets/tool-call/base-tool-props.ts` defines the `BaseToolProps` interface shared by all specialized tool renderers.
- `flitter-amp/src/widgets/tool-call/tool-output-utils.ts` provides shared extraction functions (`extractOutputText`, `extractShellOutput`, `extractDiff`, `looksLikeDiff`, `extractRawNumber`, `extractRawString`, `extractRawArray`, `joinContentText`) — pure functions with zero framework dependencies.
- `flitter-amp/src/widgets/tool-call/truncation-limits.ts` defines tiered constants (`HEADER_TRUNCATION_LIMIT`, `INPUT_TRUNCATION_LIMIT`, `PREVIEW_TRUNCATION_LIMIT`, `OUTPUT_TRUNCATION_LIMIT`, `MAX_DISPLAY_ITEMS`) and `truncateText()`/`truncateInline()` helpers.
- `flitter-amp/src/utils/raw-input.ts` provides `asString()`, `asOptionalString()`, `asNumber()`, `pickString()`, `pickNumber()`, `asArray()`, `isRecord()`, `isTodoEntry()`, `extractContentText()`, `extractAllContentText()` — all pure, zero-dependency utility functions consumed by every tool renderer.
- `flitter-cli/src/state/types.ts` already defines `ToolCallItem` and `ToolCallResult` with the same structural shape as flitter-amp's types, so no type porting is needed.

## Implementation Steps

### Step 1: Port raw-input utilities to `flitter-cli`

Create `packages/flitter-cli/src/utils/raw-input.ts`.

Copy all functions from `flitter-amp/src/utils/raw-input.ts`:
- `asString()`, `asOptionalString()`, `asNumber()` — primitive extraction
- `pickString()`, `pickNumber()` — record-level extraction
- `asArray()`, `isRecord()`, `isTodoEntry()` — array/object guards
- `extractContentText()`, `extractAllContentText()` — content text extraction

No changes needed — these are pure functions with zero external dependencies.

### Step 2: Port truncation-limits to `flitter-cli`

Create `packages/flitter-cli/src/widgets/tool-call/truncation-limits.ts`.

Copy all constants and helper functions from `flitter-amp/src/widgets/tool-call/truncation-limits.ts`:
- `HEADER_TRUNCATION_LIMIT` (80), `INPUT_TRUNCATION_LIMIT` (120), `PREVIEW_TRUNCATION_LIMIT` (500), `OUTPUT_TRUNCATION_LIMIT` (2000)
- `MAX_DISPLAY_ITEMS` (10)
- `TRUNCATION_SUFFIX`, `INLINE_TRUNCATION_SUFFIX`
- `truncateText()`, `truncateInline()`

No changes needed — all pure functions, no external dependencies.

### Step 3: Port tool-output-utils to `flitter-cli`

Create `packages/flitter-cli/src/widgets/tool-call/tool-output-utils.ts`.

Copy all functions from `flitter-amp/src/widgets/tool-call/tool-output-utils.ts`:
- `joinContentText()` — joins content array text
- `extractOutputText()` — generic text extraction with rawOutput priority
- `extractShellOutput()` — bash-specific stdout/stderr extraction
- `looksLikeDiff()`, `extractDiff()` — diff detection and extraction
- `extractRawNumber()`, `extractRawString()`, `extractRawArray()` — metadata field extraction

Adapt imports:
- `ToolCallResult` from `../../state/types` (instead of `../../acp/types`)
- `extractContentText` from `../../utils/raw-input` (ported in Step 1)
- `asString` from `../../utils/raw-input` (ported in Step 1)
- Truncation constants from `./truncation-limits` (ported in Step 2)

### Step 4: Port resolve-tool-name to `flitter-cli`

Create `packages/flitter-cli/src/widgets/tool-call/resolve-tool-name.ts`.

Copy all exports from `flitter-amp/src/widgets/tool-call/resolve-tool-name.ts`:
- `TOOL_NAME_MAP` — the complete name normalization map (covers Amp CLI, Coco ACP, and MCP variants)
- `resolveToolName()` — extracts canonical tool name from title/kind
- `resolveToolDisplayName()` — maps through TOOL_NAME_MAP for display
- `setCwd()`, `shortenPath()` — path shortening for header details
- `extractTitleDetail()` — strips leading tool name from title

Adapt imports:
- `ToolCallItem` from `../../state/types` (instead of `../../acp/types`)
- `path` from `node:path` (unchanged)

### Step 5: Port base-tool-props to `flitter-cli`

Create `packages/flitter-cli/src/widgets/tool-call/base-tool-props.ts`.

Copy the `BaseToolProps` interface and `ToolCallItem` re-export from `flitter-amp/src/widgets/tool-call/base-tool-props.ts`.

Adapt imports:
- `ToolCallItem` from `../../state/types` (instead of `../../acp/types`)

### Step 6: Create barrel export

Create `packages/flitter-cli/src/widgets/tool-call/index.ts`.

Export all infrastructure modules:
```typescript
export type { BaseToolProps } from './base-tool-props';
export * from './truncation-limits';
export * from './tool-output-utils';
export { TOOL_NAME_MAP, resolveToolName, resolveToolDisplayName, setCwd, shortenPath, extractTitleDetail } from './resolve-tool-name';
```

Tool widget exports will be added in Plans 18-02, 18-03, and 18-04 as they are created.

## Verification

- [ ] `resolveToolName({ title: 'Read /path/to/file', kind: 'read' })` returns `'Read'`
- [ ] `resolveToolDisplayName({ title: 'bash', kind: '' })` returns `'Bash'` (mapped through TOOL_NAME_MAP)
- [ ] `TOOL_NAME_MAP['execute_command']` === `'Bash'`
- [ ] `extractOutputText(result)` extracts rawOutput string, then JSON-stringifies objects, then joins content[]
- [ ] `extractShellOutput(result)` handles stdout/stderr structured objects
- [ ] `looksLikeDiff('--- a\n+++ b\n@@')` returns true
- [ ] `truncateText('a'.repeat(3000))` returns 2000 chars + truncation suffix
- [ ] `pickString({ command: 'ls' }, ['command', 'cmd'])` returns `'ls'`
- [ ] All ported modules import from `flitter-cli` paths, not `flitter-amp`
- [ ] TypeScript compilation passes for all new files

## Decisions

- **raw-input utilities live in `src/utils/`**, not `src/widgets/tool-call/` — they are generic extraction helpers used beyond tool rendering.
- **Tool-call widgets get their own subdirectory** (`src/widgets/tool-call/`) mirroring the flitter-amp structure for easy reference.
- **TOOL_NAME_MAP copied in full** — all 30+ entries including Coco ACP variants. We do not prune entries even if flitter-cli's Anthropic provider never emits them, because the fallback rendering (TOOL-04) must handle any tool name gracefully.
- **No theme references in this plan** — all utility code is theme-independent. Theme colors are introduced in Plans 18-02 and 18-03.
