# Plan 18-02: ToolHeader, GenericToolCard, ToolCallWidget Dispatch, and ChatView Integration

**Phase:** 18 — Full Tool Workflow Parity
**Requirements:** TOOL-01, TOOL-02, TOOL-04, TOOL-06 (partial)
**Depends on:** Plan 18-01 (tool infrastructure modules)
**Status:** Pending
**Estimated new tests:** 0 (widget rendering; visual verification and Phase 22 tests)

## Goal

Create the core tool rendering layer: ToolHeader (status icon + spinner + expand/collapse), GenericToolCard (fallback renderer for all unknown tools), and ToolCallWidget (top-level dispatch switch). Wire ToolCallWidget into ChatView's `buildAssistantTurnWidget()`, replacing the dim placeholder text with real tool rendering. After this plan, every tool call renders with a header, status indicator, and expandable body — even tools that don't have specialized renderers yet.

## Context

- `flitter-amp/src/widgets/tool-call/tool-header.ts` is a StatefulWidget with BrailleSpinner for in-progress status, MouseRegion for click-to-toggle, and extended status color mapping (completed/failed/in_progress/pending/cancelled/queued/blocked).
- `flitter-amp/src/widgets/tool-call/generic-tool-card.ts` is a StatelessWidget rendering StickyHeader (header=ToolHeader, body=Column of input/locations/diff-or-output sections). Acts as the fallback for all unrecognized tools.
- `flitter-amp/src/widgets/tool-call/tool-call-widget.ts` is the top-level dispatch switch routing 35+ tool names to specialized renderers or GenericToolCard fallback.
- `flitter-cli/src/widgets/chat-view.ts` lines 289-301 currently render tool calls as dim `[tool: {title}]` placeholder text. This plan replaces that with ToolCallWidget dispatch.
- `flitter-cli` does not yet have an icon registry or AmpThemeProvider. Colors will use direct Color constants (Color.green, Color.red, Color.cyan, etc.) without theme indirection. Phase 20 adds the theme system.
- `flitter-core/src/utilities/braille-spinner.ts` provides `BrailleSpinner` (already exists in the framework).
- `flitter-core/src/widgets/mouse-region.ts` provides `MouseRegion` for click handling (already exists).
- `flitter-core/src/widgets/sticky-header.ts` provides `StickyHeader` (already used in ChatView).

## Implementation Steps

### Step 1: Create status icon helper

Create `packages/flitter-cli/src/widgets/tool-call/tool-icons.ts`.

Define a `toolStatusIcon(status)` function that maps tool status to Unicode symbols:
- `'completed'` -> `'\u2713'` (checkmark)
- `'failed'` -> `'\u2717'` (cross)
- `'in_progress'` -> `'\u25CB'` (circle — spinner replaces this visually)
- `'pending'` -> `'\u25CB'` (circle)
- Default -> `'\u25CB'`

Also define `todoStatusIcon(status)` for the TodoListTool (Plan 18-03):
- `'pending'` -> `'\u25CB'` (empty circle)
- `'in_progress'` -> `'\u25D4'` (half circle)
- `'completed'` -> `'\u2713'` (checkmark)
- `'cancelled'` -> `'\u2717'` (cross)

And a generic `arrowIcon` -> `'\u2192'` for web search results.

These are simple string lookups with no external dependencies.

### Step 2: Port ToolHeader to `flitter-cli`

Create `packages/flitter-cli/src/widgets/tool-call/tool-header.ts`.

Port the `ToolHeader` StatefulWidget from flitter-amp with these adaptations:
- **Remove AmpThemeProvider dependency.** Replace `theme?.app.toolRunning ?? Color.blue` with direct `Color.blue`, etc.
- **Replace `toolStatusIcon()` import** with the local `tool-icons.ts` helper from Step 1.
- **Keep BrailleSpinner** — import from `flitter-core/src/utilities/braille-spinner`.
- **Keep MouseRegion** for `onToggle` click handling.
- **Keep the full status color mapping** (completed=green, failed=red, in_progress=blue, pending=yellow, cancelled=yellow, queued=brightBlack, blocked=red).
- **Keep normalizeInput()** for detail string truncation (80 chars).

The ToolHeader interface:
```typescript
interface ToolHeaderProps {
  name: string;
  status: ToolCallItem['status'];
  details?: string[];
  children?: Widget[];
  onToggle?: () => void;
}
```

### Step 3: Port GenericToolCard to `flitter-cli`

- **Keep streaming output handling** — show tail with block cursor during streaming, head with truncation after completion.

- Import extraction utilities from `./tool-output-utils` and `./truncation-limits`.

### Step 5: Wire ToolCallWidget into ChatView

Modify `packages/flitter-cli/src/widgets/chat-view.ts`.

Replace the tool call placeholder loop (lines 289-301) in `buildAssistantTurnWidget()`:

**Before:**
```typescript
for (const toolCall of turn.toolCalls) {
  bodyChildren.push(
    new Text({
      text: new TextSpan({
        text: `[tool: ${toolCall.title}]`,
        style: new TextStyle({
          foreground: Color.brightBlack,
          dim: true,
        }),
      }),
    }),
  );
}
```

**After:**
```typescript
for (const toolCall of turn.toolCalls) {
  bodyChildren.push(
    new ToolCallWidget({
      toolCall,
      isExpanded: !toolCall.collapsed,
    }),
  );
}
```

Add the import for `ToolCallWidget` from `./tool-call/tool-call-widget`.

This is a minimal, surgical change — the layout, spacing, and Column structure remain identical.

## Decisions

- **No theme indirection in this plan.** Colors use direct `Color.*` constants. Phase 20 introduces the theme provider; at that point, ToolHeader and GenericToolCard gain theme lookups.
- **No Markdown or DiffView rendering.** Output text is rendered as plain `Text` widgets. Phase 19 introduces Markdown and DiffView. The GenericToolCard structure is already prepared for swapping Text -> Markdown/DiffView when Phase 19 lands.
- **ToolCallWidget accepts but ignores childWidgets initially.** The prop exists for Plan 18-04's TaskTool nesting. GenericToolCard ignores it. This avoids a breaking change when nesting is added.
- **Expand/collapse state reads from `toolCall.collapsed` field.** The existing `ToolCallItem.collapsed` boolean from Phase 14's types.ts is the source of truth. No local collapsed state management needed — the session state machine owns it.
