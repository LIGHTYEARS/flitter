# Plan 18-04: Handoff Tool, Task/Sub-Agent Tool, Nested Delegation, and Permission Integration

**Phase:** 18 — Full Tool Workflow Parity
**Requirements:** TOOL-06, TOOL-07, TOOL-08
**Depends on:** Plan 18-02 (ToolHeader, GenericToolCard, ToolCallWidget), Plan 18-03 (specialized renderers in dispatch)
**Status:** Pending
**Estimated new tests:** 0 (widget rendering; visual verification and Phase 22 tests)

## Goal

Create the two remaining specialized renderers — HandoffTool and TaskTool — that handle the complex workflow patterns unique to agent delegation. Wire nested tool-tree rendering for Task/sub-agent flows where child tool calls appear indented under their parent. Update ToolCallWidget dispatch to route handoff, Task, oracle, code_review, librarian, and `sa__`/`tb__`-prefixed tools. Verify that the permission flow from Phase 17 (TOOL-06) integrates correctly with tool status transitions.

## Context

- `flitter-amp/src/widgets/tool-call/handoff-tool.ts` is a **StatefulWidget** with a 700ms blink animation (setInterval + setState) showing a waiting indicator during `in_progress` status. The blink alternates between theme green and theme muted. When complete, it shows the handoff result output.
- `flitter-amp/src/widgets/tool-call/task-tool.ts` is a StatelessWidget that accepts `childWidgets?: Widget[]` — these are pre-built ToolCallWidgets for child tool calls belonging to the same Task delegation. When childWidgets are provided, they render indented under the Task header. Without childWidgets, a summary text is shown.
- `flitter-cli/src/state/types.ts` already defines `ToolCallItem.parentToolCallId?: string` for identifying nested tool calls that belong to a parent Task. The ChatView turn model has `turn.toolCalls` as a flat array; nesting is reconstructed at render time.
- Phase 17's `AppState.requestPermission()` handles the permission dialog modal for tool execution approval. This is already wired via `OverlayManager`. TOOL-06 requires that permission, progress, completion, interruption, and failure states remain visible — this is handled by ToolHeader's status colors and GenericToolCard's output display from Plan 18-02.
- The `ToolCallItem.status` field already tracks `'pending' | 'in_progress' | 'completed' | 'failed'`, and `SessionState.updateToolCall()` transitions status with listener notification. No additional permission state plumbing is needed — the permission dialog resolves before the tool call moves from pending to in_progress.

### Step 3: Build nested tool-tree rendering in ChatView

Modify `packages/flitter-cli/src/widgets/chat-view.ts`.

The flat `turn.toolCalls` array contains both parent and child tool calls. Child tool calls have `parentToolCallId` set to their parent's `toolCallId`. The rendering must:

1. **Partition tool calls** into root-level (no parentToolCallId) and children (has parentToolCallId).
2. **Build a parent->children map**: `Map<string, ToolCallItem[]>` keyed by parentToolCallId.
3. **For each root tool call:**
   - Look up children in the map.
   - If children exist, recursively build child ToolCallWidgets and pass them as `childWidgets` to the parent's ToolCallWidget.
   - If no children, render the parent ToolCallWidget without childWidgets.

Update the tool call rendering section in `buildAssistantTurnWidget()`:

```typescript
// Build parent->children map for nested tool-tree rendering
const childMap = new Map<string, ToolCallItem[]>();
const rootToolCalls: ToolCallItem[] = [];
for (const tc of turn.toolCalls) {
  if (tc.parentToolCallId) {
    const siblings = childMap.get(tc.parentToolCallId) ?? [];
    siblings.push(tc);
    childMap.set(tc.parentToolCallId, siblings);
  } else {
    rootToolCalls.push(tc);
  }
}

for (const toolCall of rootToolCalls) {
  const children = childMap.get(toolCall.toolCallId);
  const childWidgets = children?.map(child =>
    new ToolCallWidget({ toolCall: child, isExpanded: !child.collapsed })
  );
  bodyChildren.push(
    new ToolCallWidget({
      toolCall,
      isExpanded: !toolCall.collapsed,
      childWidgets,
    }),
  );
}
```

This is a flat-to-tree reconstruction at render time. It supports one level of nesting (Task -> child tools). Deeper nesting (Task -> Task -> child) would require recursive map lookups, but the Amp reference only supports one level, so we match that behavior.

### Step 5: Verify permission integration (TOOL-06)

No new code needed. Verify the existing flow:

1. Provider emits `StreamToolCallStart` -> SessionState.addToolCall() creates ToolCallItem with status `'pending'`.
2. If permission needed, PromptController calls `AppState.requestPermission()` -> OverlayManager shows PermissionDialog.
3. User approves/denies -> PromptController proceeds or aborts.
4. If approved, tool moves to `'in_progress'` -> ToolHeader shows blue + BrailleSpinner.
5. Provider emits `StreamToolCallEnd` -> SessionState.updateToolCall() sets `'completed'` or `'failed'` -> ToolHeader shows green/red.
6. If user cancels (Ctrl+C), SessionState.cancelStream() -> all in-flight tools remain in their last status.

The ToolHeader status colors (Plan 18-02) already handle all these states. The GenericToolCard and specialized renderers display output for completed tools and error information for failed tools. No additional wiring is needed beyond what Plans 18-01 through 18-03 established.

## Decisions

- **One level of nesting only.** The Amp reference implementation also uses one level (Task -> child tools). Deeper nesting (Task -> Task -> ...) is not supported. If a child tool call itself is a Task with children, those grandchildren render as summary text inside the child Task, not as further-nested widgets.
- **Orphan child tool calls render at root level.** If a tool call has `parentToolCallId` but the parent is not in the current turn's toolCalls array (e.g., parent was in a different turn), the child renders as a root-level entry. This prevents invisible tool calls.
- **HandoffTool blink uses direct Color values, not theme tokens.** Green -> `Color.green`, dim -> `Color.brightBlack`. Phase 20 adds theme indirection.
- **Permission integration is verified, not implemented.** The existing Phase 17 permission flow (AppState.requestPermission -> OverlayManager -> PermissionDialog) already handles TOOL-06. This plan only verifies the integration by confirming status transitions are visible through ToolHeader.
