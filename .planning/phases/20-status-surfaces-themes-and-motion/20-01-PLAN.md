# Plan 20-01: Theme System

**Phase:** 20 — Status Surfaces, Themes, and Motion
**Requirements:** STAT-01, STAT-02
**Wave:** 1 (no dependencies on other plans in this phase)

## Goal

Introduce a theme system for flitter-cli that replaces the hardcoded `Color.*` constants scattered across 22+ widget files. Port the `AmpBaseTheme`/`AmpAppColors`/`AmpTheme` type system, `deriveAppColors()` derivation logic, `InheritedWidget`-based provider, and all 7 built-in themes (dark, light, catppuccin-mocha, solarized-dark, solarized-light, gruvbox-dark, nord) from flitter-amp. Add a `theme` field to user config and wrap the app layout in `CliThemeProvider`.

## Current State

1. **All widgets use hardcoded `Color.*` constants.** Comments throughout (e.g., `streaming-cursor.ts:6`, `input-area.ts:21`, `shortcut-help-overlay.ts:13`, `file-picker.ts:25`) explicitly note "Phase 20 adds theme."
2. **flitter-amp has a complete theme system:** `AmpBaseTheme` (16 base colors + syntaxHighlight), `AmpAppColors` (50+ semantic tokens), `AmpTheme` composite, `deriveAppColors()` derivation, `AmpThemeProvider` InheritedWidget, and 7 built-in themes.
3. **`AppConfig` (config.ts)** has `cwd`, `editor`, `logLevel`, `logRetentionDays`, `apiKey`, `model` but no `theme` field.
4. **`AppShell` (app-shell.ts:458)** has a `// StatusBar placeholder — Phase 20` comment and no `CliThemeProvider` wrapper.
5. **flitter-core `InheritedWidget`** and `BuildContext.dependOnInheritedWidgetOfExactType()` are available and proven (used by `AmpThemeProvider` in flitter-amp).

## Approach

Port the theme type system, derivation function, and InheritedWidget provider from flitter-amp, renaming `Amp*` → `Cli*` prefixes for flitter-cli ownership. Copy all 7 theme files with exact RGB values. Add `theme: string` to config with `--theme` CLI flag. Wrap `AppShell.build()` output in `CliThemeProvider`.

### Why Cli prefix

flitter-cli is a standalone app, not a wrapper around flitter-amp. Using `Cli*` prefix signals clear ownership while maintaining structural parity for easy diffing.

### File 2: `packages/flitter-cli/src/themes/index.ts` (NEW)

**Action:** Create new file.

Port from `flitter-amp/src/themes/index.ts`:

1. **`CliThemeProvider`** — `InheritedWidget` subclass with `theme: CliTheme` property. Static `of(context)` (throws) and `maybeOf(context)` (returns undefined) lookup methods. `updateShouldNotify()` with deep equality check.
2. **`deriveAppColors(base: CliBaseTheme): CliAppColors`** — identical logic to flitter-amp's `deriveAppColors()`, using `Color.ansi256(8)` for dim gray, mode-sensitive `smartModeColor`/`rushModeColor`, and ANSI named colors for all semantic roles.
3. **`createCliTheme(base: CliBaseTheme): CliTheme`** — combines base + derived app colors.
4. **`cliThemes: Readonly<Record<string, CliBaseTheme>>`** — registry mapping 7 theme names to their `CliBaseTheme` definitions.
5. **Deep equality helpers** — `cliThemeEquals()`, `cliBaseThemeEquals()`, `cliAppColorsEquals()` for `updateShouldNotify`.
6. **Re-exports** of all types and theme constants.

Import paths use `../../../flitter-core/src/...` pattern consistent with flitter-cli widget files.

   Before:
   ```typescript
   return new FocusScope({
     autofocus: true,

     child: withOverlays,
   });
   ```

   After:
   ```typescript
   const baseTheme = cliThemes[this.widget.themeName] ?? cliThemes['dark']!;

   return new CliThemeProvider({
     theme,
     child: new FocusScope({
       autofocus: true,
       onKey: (event: KeyEvent): KeyEventResult => this._handleKey(event),
       child: withOverlays,
     }),
   });
   ```

4. Wire `themeName` from `AppConfig.theme` through `startAppShell()` into the `AppShell` widget.

4. **STAT-02 — Config theme selection**: `flitter-cli --theme catppuccin-mocha` sets the active theme. Invalid theme name falls back to `'dark'` with a warning.
5. **STAT-02 — Default theme**: Launching without `--theme` uses `'dark'` theme.
