# Plan 20-03: Wire Themes Through Widgets

**Phase:** 20 — Status Surfaces, Themes, and Motion
**Requirements:** STAT-04
**Wave:** 2 (depends on 20-01 for CliThemeProvider and theme types)

## Goal

Replace all hardcoded `Color.*` constants in flitter-cli widgets with theme-derived colors read from `CliThemeProvider.maybeOf(context)`. Every widget that renders colored text or borders should use semantic theme tokens (base or app) with a fallback to the current hardcoded value for safety. After this plan, switching themes via `--theme` visually affects the entire UI.

## Current State

1. **Plan 20-01** (prerequisite) provides `CliThemeProvider`, `CliTheme`, `CliBaseTheme`, `CliAppColors` types, and wraps AppShell in a provider.
2. **22 widget files** use `Color.*` constants (109 total occurrences across all files per grep).
3. **flitter-amp** already performs this migration — each widget reads colors via `AmpThemeProvider.maybeOf(context)` with `??` fallbacks. This plan ports the same pattern to flitter-cli.
4. **The pattern is consistent:** `const theme = CliThemeProvider.maybeOf(context);` then `theme?.base.success ?? Color.green` or `theme?.app.toolSuccess ?? Color.green`.

## Approach

Edit each widget file to:
1. Add import for `CliThemeProvider` from the appropriate relative path.
2. In `build(context)`, call `const theme = CliThemeProvider.maybeOf(context);` (requires `context` parameter — currently some `build()` signatures use `_context`; update to `context`).
3. Replace each `Color.*` literal with the corresponding theme token + fallback.

The fallback pattern `theme?.base.X ?? Color.Y` ensures the widget still renders correctly if no theme provider exists in the tree (e.g., in isolated tests).

### Color Mapping Table

| Hardcoded Color | Theme Token | Role |
|---|---|---|
| `Color.green` | `theme?.base.success` | Success states, done icons |
| `Color.yellow` | `theme?.base.warning` | Warning, interrupted, pending |
| `Color.red` | `theme?.base.destructive` | Error, failed |
| `Color.blue` | `theme?.base.info` / `theme?.app.keybind` | Keybinds, info, in_progress |
| `Color.magenta` | `theme?.base.accent` | Streaming accent, thinking |
| `Color.cyan` | `theme?.app.userMessage` / `theme?.app.fileReference` | User message, file refs, info |
| `Color.brightBlack` | `theme?.base.mutedForeground` | Muted/dim text, borders |
| `Color.brightCyan` | `theme?.base.secondary` | Streaming user border |
| `Color.defaultColor` | `theme?.base.foreground` | Default foreground text |
| `Color.white` | `theme?.base.foreground` | Explicit white foreground |

### UAT Criteria

1. **STAT-04 — Default theme visual parity**: Launch with `--theme dark` (default). All widget colors should be visually identical to the pre-theme hardcoded appearance (the `deriveAppColors()` ANSI mappings match the previous `Color.*` constants).
2. **STAT-04 — Theme switching**: Launch with `--theme catppuccin-mocha`. Observe:
   - User message borders use Catppuccin's secondary color instead of default cyan.
   - Thinking block streaming uses Catppuccin's accent (purple-pink) instead of ANSI magenta.
   - Diff card borders use Catppuccin's muted foreground instead of brightBlack.
   - Status bar text uses Catppuccin's info color instead of default blue.
3. **STAT-04 — Light theme**: Launch with `--theme light`. Verify text is readable on a light terminal background.
4. **STAT-04 — Fallback safety**: If `CliThemeProvider` is removed from the tree (e.g., in a unit test rendering a widget in isolation), all widgets still render correctly with fallback `Color.*` values.

### Regression Checks

- All existing visual tests (if any) still pass — colors may shift slightly if tests assert specific ANSI codes, but semantic meaning is preserved.
- Widget layout (padding, spacing, column structure) is unchanged — only `Color` values change.
- Interactive behavior (focus, keyboard, mouse) is unchanged.
- Tool call expand/collapse, spinner animation, and streaming cursor blink all unchanged.
