**Step 1: Create `packages/flitter-cli/src/provider/pi-ai-provider.ts`** — the adapter class.

Per D-00, D-18, D-19 — this is a thin adapter that delegates to pi-ai's stream().

```typescript
import { stream as piAiStream, getModel, getEnvApiKey, calculateCost } from '@mariozechner/pi-ai';
import type { Model, Api, AssistantMessageEvent, Context } from '@mariozechner/pi-ai';
import type { Provider, PromptOptions, ProviderId, ProviderCapabilities } from './provider';
import type { StreamEvent, ProviderMessage, UsageInfo } from '../state/types';
import { log } from '../utils/logger';
```

**PiAiProvider class:**
- Constructor takes `(piModel: Model<Api>, providerId: ProviderId, providerName: string, apiKey: string)`.
- `readonly piModel: Model<Api>` — exposed per D-18.
- `readonly id: ProviderId` — from constructor.
- `readonly name: string` — from constructor.
- `readonly model: string` — `piModel.id`.
- `readonly capabilities: ProviderCapabilities` — derived from piModel:
  - `vision: piModel.input.includes('image')`
  - `functionCalling: true` (pi-ai only includes tool-capable models)
  - `streaming: true`
  - `systemPrompt: true`
- Private `_abort: AbortController | null = null`.
- `cancelRequest()` — aborts current controller.

**sendPrompt() method:**
- Creates new AbortController, combines with `options.abortSignal` if present.
- Builds pi-ai `Context` from `ProviderMessage[]` + `options.systemPrompt`.
  The Context mapping:
  - `options.systemPrompt` → `{ role: 'system', content: systemPrompt }` in context messages
  - Each ProviderMessage maps to pi-ai message format:
    - `role: 'user' | 'assistant'` with text content
    - `role: 'user'` for tool results (tool_result blocks)
    - `role: 'assistant'` for tool_use blocks
  - Tools from `options.tools` map to pi-ai tool definitions
- Calls `piAiStream(this.piModel, context, { apiKey: this._apiKey, signal, maxTokens: options.maxTokens })`.
- Iterates pi-ai `AssistantMessageEvent` stream and yields flitter-cli `StreamEvent` via `mapPiAiEvent()`.

**mapPiAiEvent() — private async generator helper (per D-19):**
Maps pi-ai events to StreamEvent. The mapping table from CONTEXT.md:

| pi-ai event | flitter-cli StreamEvent |
|---|---|
| `{ type: 'text_delta', delta }` | `{ type: 'text_delta', text: delta }` |
| `{ type: 'thinking_start' }` | (no direct mapping needed — thinking_delta handles it) |
| `{ type: 'thinking_delta', delta }` | `{ type: 'thinking_delta', text: delta }` |
| `{ type: 'thinking_end' }` | (no event needed) |
| `{ type: 'toolcall_start', toolCall }` | `{ type: 'tool_call_start', toolCallId: toolCall.id, name: toolCall.name, title: toolCall.name, kind: 'tool' }` |
| `{ type: 'toolcall_delta', ... }` | `{ type: 'tool_call_input_delta', toolCallId, partialJson }` |
| `{ type: 'toolcall_end', toolCall }` | `{ type: 'tool_call_ready', toolCallId: toolCall.id, name: toolCall.name, input: toolCall.input }` |
| `{ type: 'done', reason, message }` | `{ type: 'usage_update', usage }` (from message.usage) THEN `{ type: 'message_complete', stopReason: mapStopReason(reason) }` |
| `{ type: 'error', reason, error }` | `{ type: 'error', error: { message: reason, code: 'PI_AI_ERROR', retryable: true } }` |
| `{ type: 'start' }` | (skip — no flitter-cli equivalent) |

`mapStopReason()`: Maps pi-ai stop reasons to flitter-cli reasons:
- `'end_turn'` / `'stop'` → `'end_turn'`
- `'tool_use'` → `'tool_use'`
- `'max_tokens'` → `'max_tokens'`
- Other → pass through as string

For the `done` event, extract usage from `message.usage`:
```typescript
const usage: UsageInfo = {
  size: this.piModel.contextWindow,
  used: (message.usage?.inputTokens ?? 0) + (message.usage?.outputTokens ?? 0),
  inputTokens: message.usage?.inputTokens,
  outputTokens: message.usage?.outputTokens,
  cost: message.usage ? {
    amount: calculateCost(this.piModel, message.usage),
    currency: '$',
  } : null,
};
```

**Rewritten `createProvider(config: ProviderConfig): Provider`:**
1. Resolve the pi-ai provider key from `PROVIDER_MAP[config.id]` (fall back to `config.id` for unknown providers).
2. Resolve model ID: `config.model ?? DEFAULT_MODELS[config.id] ?? 'claude-sonnet-4-20250514'`.
3. Resolve API key by priority:
   a. `config.apiKey` if provided
   b. For OAuth providers (chatgpt-codex, copilot, antigravity): load from `loadToken(config.id)?.accessToken`
   c. `getEnvApiKey(piProviderKey)` from pi-ai
   d. Throw descriptive error if none found
4. Look up the model: `getModel(piProviderKey, modelId)`.
   - If `getModel` throws (unknown model), try `getModels(piProviderKey)` and find closest match or use first model.
5. For OAuth providers, validate token exists and throw auth instructions if not (preserve existing error messages from current factory.ts).
6. For copilot: set baseUrl override via model object if `config.auth?.accountId` provides a proxy URL.
7. Return `new PiAiProvider(model, config.id, displayName, apiKey)`.

**Rewritten `autoDetectProvider(): ProviderConfig | null`** (per D-16):
```typescript
export function autoDetectProvider(): ProviderConfig | null {
  // Check all pi-ai providers for env keys
  const piProviders = getProviders();
  for (const p of piProviders) {
    const key = getEnvApiKey(p);
    if (key) {
      // Map pi-ai provider back to flitter-cli ProviderId
      const flitterId = REVERSE_PROVIDER_MAP[p] ?? p;
      return { id: flitterId as ProviderId, apiKey: key };
    }
  }
  // Check OAuth tokens
  if (hasValidToken('chatgpt-codex')) return { id: 'chatgpt-codex' };
  if (hasValidToken('copilot')) return { id: 'copilot' };
  if (hasValidToken('antigravity')) return { id: 'antigravity' };
  return null;
}
```
